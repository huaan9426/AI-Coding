# 第16章：多集群管理与联邦

## 章节概述

随着企业云原生应用规模的扩大，单一Kubernetes集群往往无法满足以下需求：
- **高可用性**：避免单点故障，实现跨区域容灾
- **地理分布**：就近服务，降低延迟
- **资源隔离**：不同环境（开发/测试/生产）、不同租户的隔离
- **合规要求**：数据主权、监管要求
- **多云战略**：避免供应商锁定，利用不同云厂商优势

本章将深入探讨Kubernetes多集群管理的架构模式、技术方案和最佳实践。

**本章内容：**
- 16.1 多集群架构模式
- 16.2 KubeFed联邦集群
- 16.3 Cluster API
- 16.4 多集群服务网格
- 16.5 跨集群负载均衡
- 16.6 实战：构建多云Kubernetes平台

## 16.1 多集群架构模式

### 16.1.1 为什么需要多集群

#### 单集群的局限性

**1. 可用性风险**
```
单集群故障影响范围：
┌─────────────────────────────────┐
│     单一Kubernetes集群           │
│  ┌──────────┐  ┌──────────┐    │
│  │ Master 1 │  │ Master 2 │    │
│  └──────────┘  └──────────┘    │
│  ┌──────────────────────────┐  │
│  │    所有应用和服务         │  │
│  └──────────────────────────┘  │
└─────────────────────────────────┘
         │
         ▼ 集群故障
    ❌ 所有服务不可用
```

**2. 性能瓶颈**
- 单集群规模限制（通常5000节点上限）
- 跨地域延迟高
- 资源竞争激烈

**3. 管理复杂度**
- 多租户隔离困难
- 升级影响范围大
- 资源配额管理复杂

**4. 合规性挑战**
- 数据主权要求
- 监管合规性
- 审计追踪

#### 多集群的优势

**1. 高可用性**
```
多集群容灾架构：
┌──────────────┐      ┌──────────────┐
│  集群A (主)   │      │  集群B (备)   │
│  Region: US  │◄────►│  Region: EU  │
│  ┌────────┐  │      │  ┌────────┐  │
│  │ 应用A  │  │      │  │ 应用A  │  │
│  └────────┘  │      │  └────────┘  │
└──────────────┘      └──────────────┘
         │                    │
         └────────┬───────────┘
                  ▼
          ✅ 任一集群故障
             服务仍可用
```

**2. 地理分布**
- 就近服务，降低延迟
- 满足数据本地化要求
- 提升用户体验

**3. 资源优化**
- 弹性扩展到多个集群
- 利用不同云厂商优势
- 成本优化

**4. 环境隔离**
- 开发/测试/生产环境分离
- 多租户隔离
- 安全边界清晰

### 16.1.2 多集群架构模式

#### 模式1：独立集群（Independent Clusters）

**架构特点：**
- 每个集群完全独立
- 无集群间通信
- 应用独立部署

**架构图：**
```
┌─────────────────┐  ┌─────────────────┐  ┌─────────────────┐
│   集群A (Dev)    │  │  集群B (Test)    │  │  集群C (Prod)   │
│                 │  │                 │  │                 │
│  ┌───────────┐  │  │  ┌───────────┐  │  │  ┌───────────┐  │
│  │  应用部署  │  │  │  │  应用部署  │  │  │  │  应用部署  │  │
│  └───────────┘  │  │  └───────────┘  │  │  └───────────┘  │
│                 │  │                 │  │                 │
└─────────────────┘  └─────────────────┘  └─────────────────┘
        │                    │                    │
        └────────────────────┴────────────────────┘
                             │
                    ┌────────▼────────┐
                    │   CI/CD Pipeline │
                    │  独立部署到各集群  │
                    └─────────────────┘
```

**适用场景：**
- 环境隔离（开发/测试/生产）
- 多租户完全隔离
- 简单的多集群需求

**优点：**
- 架构简单，易于理解
- 故障隔离彻底
- 管理独立，互不影响

**缺点：**
- 资源利用率低
- 管理成本高
- 无法跨集群调度

**实现示例：**
```yaml
# 使用kubectl context管理多集群
# 配置集群A
kubectl config set-cluster cluster-a \
  --server=https://cluster-a.example.com \
  --certificate-authority=ca-a.crt

kubectl config set-context cluster-a \
  --cluster=cluster-a \
  --user=admin-a

# 配置集群B
kubectl config set-cluster cluster-b \
  --server=https://cluster-b.example.com \
  --certificate-authority=ca-b.crt

kubectl config set-context cluster-b \
  --cluster=cluster-b \
  --user=admin-b

# 切换集群
kubectl config use-context cluster-a
kubectl apply -f app.yaml

kubectl config use-context cluster-b
kubectl apply -f app.yaml
```

#### 模式2：主从集群（Hub-Spoke）

**架构特点：**
- 一个中心集群（Hub）管理多个工作集群（Spoke）
- 中心集群负责策略分发、配置管理
- 工作集群执行实际工作负载

**架构图：**
```
                    ┌─────────────────────┐
                    │   Hub Cluster       │
                    │  (管理控制平面)      │
                    │  ┌───────────────┐  │
                    │  │ 策略管理       │  │
                    │  │ 配置分发       │  │
                    │  │ 监控聚合       │  │
                    │  └───────────────┘  │
                    └──────────┬──────────┘
                               │
              ┌────────────────┼────────────────┐
              │                │                │
              ▼                ▼                ▼
    ┌─────────────────┐ ┌─────────────────┐ ┌─────────────────┐
    │ Spoke Cluster 1 │ │ Spoke Cluster 2 │ │ Spoke Cluster 3 │
    │  (工作负载)      │ │  (工作负载)      │ │  (工作负载)      │
    │  ┌───────────┐  │ │  ┌───────────┐  │ │  ┌───────────┐  │
    │  │  应用A     │  │ │  │  应用B     │  │ │  │  应用C     │  │
    │  └───────────┘  │ │  └───────────┘  │ │  └───────────┘  │
    └─────────────────┘ └─────────────────┘ └─────────────────┘
```

**适用场景：**
- 集中式管理需求
- 统一策略分发
- 多地域部署

**优点：**
- 集中管理，统一视图
- 策略一致性
- 简化运维

**缺点：**
- Hub集群成为单点
- Hub集群压力大
- 网络依赖性强

**实现示例（使用KubeFed）：**
```yaml
# Hub集群配置
apiVersion: v1
kind: Namespace
metadata:
  name: kube-federation-system

---
# 注册Spoke集群
apiVersion: core.kubefed.io/v1beta1
kind: KubeFedCluster
metadata:
  name: spoke-cluster-1
  namespace: kube-federation-system
spec:
  apiEndpoint: https://spoke-1.example.com
  caBundle: <base64-encoded-ca>
  secretRef:
    name: spoke-cluster-1-secret

---
# 联邦部署
apiVersion: types.kubefed.io/v1beta1
kind: FederatedDeployment
metadata:
  name: my-app
  namespace: default
spec:
  template:
    metadata:
      labels:
        app: my-app
    spec:
      replicas: 3
      selector:
        matchLabels:
          app: my-app
      template:
        metadata:
          labels:
            app: my-app
        spec:
          containers:
          - name: app
            image: my-app:v1.0
  placement:
    clusters:
    - name: spoke-cluster-1
    - name: spoke-cluster-2
```

#### 模式3：网格集群（Mesh）

**架构特点：**
- 所有集群地位平等
- 集群间可以直接通信
- 无中心控制节点

**架构图：**
```
    ┌─────────────────┐
    │   Cluster A     │
    │  ┌───────────┐  │
    │  │  应用A     │◄─┼────────┐
    │  └───────────┘  │        │
    └────────┬────────┘        │
             │                 │
             │                 │
    ┌────────▼────────┐        │
    │   Cluster B     │        │
    │  ┌───────────┐  │        │
    │  │  应用B     │◄─┼────────┤
    │  └───────────┘  │        │
    └────────┬────────┘        │
             │                 │
             │                 │
    ┌────────▼────────┐        │
    │   Cluster C     │        │
    │  ┌───────────┐  │        │
    │  │  应用C     │◄─┼────────┘
    │  └───────────┘  │
    └─────────────────┘

    所有集群互联互通
```

**适用场景：**
- 服务网格场景
- 微服务跨集群通信
- 无中心化需求

**优点：**
- 无单点故障
- 灵活性高
- 扩展性好

**缺点：**
- 配置复杂
- 网络开销大
- 一致性难保证

**实现示例（使用Istio多集群）：**
```yaml
# 集群A配置
apiVersion: install.istio.io/v1alpha1
kind: IstioOperator
metadata:
  name: istio-control-plane
spec:
  values:
    global:
      meshID: mesh1
      multiCluster:
        clusterName: cluster-a
      network: network-a

---
# 集群B配置
apiVersion: install.istio.io/v1alpha1
kind: IstioOperator
metadata:
  name: istio-control-plane
spec:
  values:
    global:
      meshID: mesh1
      multiCluster:
        clusterName: cluster-b
      network: network-b

---
# 跨集群服务发现
apiVersion: networking.istio.io/v1beta1
kind: ServiceEntry
metadata:
  name: service-a-from-cluster-b
spec:
  hosts:
  - service-a.default.svc.cluster.local
  location: MESH_INTERNAL
  ports:
  - number: 80
    name: http
    protocol: HTTP
  resolution: DNS
  endpoints:
  - address: service-a.cluster-b.global
    ports:
      http: 80
```

**架构特点：**

### 16.1.3 多集群网络模型

#### 网络模型1：扁平网络（Flat Network）

**特点：**
- 所有集群在同一网络平面
- Pod可以直接跨集群通信
- 需要统一的IP地址规划

**架构图：**
```
┌─────────────────────────────────────────────────────────┐
│              扁平网络 (10.0.0.0/8)                       │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  ┌──────────────────┐         ┌──────────────────┐    │
│  │  Cluster A       │         │  Cluster B       │    │
│  │  Pod CIDR:       │         │  Pod CIDR:       │    │
│  │  10.1.0.0/16     │◄───────►│  10.2.0.0/16     │    │
│  │                  │         │                  │    │
│  │  Pod: 10.1.1.10  │         │  Pod: 10.2.1.20  │    │
│  └──────────────────┘         └──────────────────┘    │
│                                                         │
└─────────────────────────────────────────────────────────┘

Pod 10.1.1.10 可以直接访问 Pod 10.2.1.20
```

**实现方式：**
- VPN隧道（WireGuard、IPSec）
- 云厂商VPC Peering
- Calico跨集群路由

**配置示例（Calico）：**
```yaml
# 集群A配置
apiVersion: projectcalico.org/v3
kind: IPPool
metadata:
  name: cluster-a-pool
spec:
  cidr: 10.1.0.0/16
  ipipMode: CrossSubnet
  natOutgoing: true

---
# 集群B配置
apiVersion: projectcalico.org/v3
kind: IPPool
metadata:
  name: cluster-b-pool
spec:
  cidr: 10.2.0.0/16
  ipipMode: CrossSubnet
  natOutgoing: true

---
# BGP Peer配置（集群A）
apiVersion: projectcalico.org/v3
kind: BGPPeer
metadata:
  name: cluster-b-peer
spec:
  peerIP: <cluster-b-node-ip>
  asNumber: 64512
```

**优点：**
- 性能最优
- 延迟最低
- 配置相对简单

**缺点：**
- IP地址规划复杂
- 网络安全风险
- 跨云厂商困难

#### 网络模型2：网关模式（Gateway）

**特点：**
- 集群间通过网关通信
- 每个集群独立的网络空间
- 网关负责流量转发和NAT

**架构图：**
```
┌──────────────────┐                    ┌──────────────────┐
│   Cluster A      │                    │   Cluster B      │
│   Pod CIDR:      │                    │   Pod CIDR:      │
│   10.1.0.0/16    │                    │   10.1.0.0/16    │
│                  │                    │                  │
│  ┌────────────┐  │                    │  ┌────────────┐  │
│  │ Pod A      │  │                    │  │ Pod B      │  │
│  │ 10.1.1.10  │  │                    │  │ 10.1.1.20  │  │
│  └─────┬──────┘  │                    │  └─────▲──────┘  │
│        │         │                    │        │         │
│        ▼         │                    │        │         │
│  ┌────────────┐  │    ┌─────────┐    │  ┌────────────┐  │
│  │  Gateway   │◄─┼───►│ Internet│◄───┼─►│  Gateway   │  │
│  │  (Ingress) │  │    └─────────┘    │  │  (Ingress) │  │
│  └────────────┘  │                    │  └────────────┘  │
└──────────────────┘                    └──────────────────┘

Pod A → Gateway A → Internet → Gateway B → Pod B
```

**实现方式：**
- Istio Gateway
- Ingress Controller
- Service Mesh Gateway

**配置示例（Istio）：**
```yaml
# 集群A的Gateway
apiVersion: networking.istio.io/v1beta1
kind: Gateway
metadata:
  name: cross-cluster-gateway
spec:
  selector:
    istio: ingressgateway
  servers:
  - port:
      number: 443
      name: tls
      protocol: TLS
    tls:
      mode: AUTO_PASSTHROUGH
    hosts:
    - "*.cluster-b.global"

---
# 集群A访问集群B的服务
apiVersion: networking.istio.io/v1beta1
kind: ServiceEntry
metadata:
  name: service-b-external
spec:
  hosts:
  - service-b.default.svc.cluster-b.global
  location: MESH_EXTERNAL
  ports:
  - number: 80
    name: http
    protocol: HTTP
  resolution: DNS
  endpoints:
  - address: gateway-b.example.com
    ports:
      http: 443

---
# 集群A的VirtualService
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: service-b-route
spec:
  hosts:
  - service-b.default.svc.cluster-b.global
  http:
  - route:
    - destination:
        host: service-b.default.svc.cluster-b.global
        port:
          number: 80
```

**优点：**
- 网络隔离性好
- 安全性高
- 支持跨云厂商

**缺点：**
- 性能开销
- 延迟增加
- 配置复杂

### 16.1.4 多集群服务发现

#### 方案1：DNS联邦

**原理：**
- 使用DNS实现跨集群服务发现
- 每个集群有独立的DNS域
- 通过DNS转发实现跨集群解析

**架构图：**
```
┌──────────────────────────────────────────────────────────┐
│                    DNS联邦                                │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  ┌────────────────┐              ┌────────────────┐    │
│  │  Cluster A     │              │  Cluster B     │    │
│  │  DNS域:        │              │  DNS域:        │    │
│  │  cluster-a.    │              │  cluster-b.    │    │
│  │  local         │              │  local         │    │
│  │                │              │                │    │
│  │  service-a.    │              │  service-b.    │    │
│  │  default.svc.  │◄────────────►│  default.svc.  │    │
│  │  cluster-a.    │   DNS转发    │  cluster-b.    │    │
│  │  local         │              │  local         │    │
│  └────────────────┘              └────────────────┘    │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

**配置示例（CoreDNS）：**
```yaml
# 集群A的CoreDNS配置
apiVersion: v1
kind: ConfigMap
metadata:
  name: coredns
  namespace: kube-system
data:
  Corefile: |
    .:53 {
        errors
        health {
           lameduck 5s
        }
        ready
        kubernetes cluster-a.local in-addr.arpa ip6.arpa {
           pods insecure
           fallthrough in-addr.arpa ip6.arpa
           ttl 30
        }
        # 转发cluster-b的DNS查询
        forward cluster-b.local <cluster-b-dns-ip>
        prometheus :9153
        forward . /etc/resolv.conf {
           max_concurrent 1000
        }
        cache 30
        loop
        reload
        loadbalance
    }
```

**优点：**
- 实现简单
- 兼容性好
- 无需修改应用

**缺点：**
- DNS缓存问题
- 故障切换慢
- 负载均衡能力弱

#### 方案2：服务网格（Service Mesh）

**原理：**
- 使用服务网格实现跨集群服务发现
- Sidecar代理处理服务发现和路由
- 控制平面同步服务信息

**架构图：**
```
┌──────────────────────────────────────────────────────────┐
│              Istio多集群服务网格                          │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  ┌────────────────┐              ┌────────────────┐    │
```

#### 数据一致性策略

**1. 最终一致性（Eventual Consistency）**

**适用场景：**
- 读多写少的场景
- 对一致性要求不高
- 可以容忍短暂不一致

**实现方式：**
```yaml
# 使用异步复制
apiVersion: v1
kind: ConfigMap
metadata:
  name: data-sync-config

  sync-strategy: "async"
  sync-interval: "30s"
  conflict-resolution: "last-write-wins"
```

**2. 强一致性（Strong Consistency）**

**适用场景：**
- 金融交易
- 库存管理
- 关键业务数据

**实现方式：**
```yaml
# 使用分布式事务
apiVersion: v1
kind: ConfigMap
metadata:
  name: data-sync-config
data:
  sync-strategy: "sync"
  transaction-mode: "2pc"  # 两阶段提交
  timeout: "5s"
```

**3. 因果一致性（Causal Consistency）**

**适用场景：**
- 社交网络
- 协作应用
- 消息系统

**实现方式：**
```yaml
# 使用向量时钟
apiVersion: v1
kind: ConfigMap

  name: data-sync-config

  sync-strategy: "causal"
  vector-clock-enabled: "true"
  conflict-resolution: "vector-clock"
```

```go
func getClusterForUser(userID string) string {
    hash := crc32.ChecksumIEEE([]byte(userID))
    clusterIndex := hash % uint32(len(clusters))
    return clusters[clusterIndex]
}
```

## 16.2 KubeFed联邦集群

### 16.2.1 KubeFed概述

#### 什么是KubeFed

KubeFed（Kubernetes Cluster Federation）是Kubernetes官方的多集群管理解决方案，它允许你从单一控制平面协调多个Kubernetes集群的配置。

**核心概念：**

```
KubeFed架构：
┌─────────────────────────────────────────────────────────┐
│              Host Cluster (控制平面)                     │
│  ┌───────────────────────────────────────────────────┐  │
│  │  KubeFed Control Plane                            │  │
│  │  ┌─────────────┐  ┌─────────────┐  ┌──────────┐ │  │
│  │  │ Federation  │  │ Scheduler   │  │ Sync     │ │  │
│  │  │ Controller  │  │ Controller  │  │Controller│ │  │
│  │  └─────────────┘  └─────────────┘  └──────────┘ │  │
│  └───────────────────────────────────────────────────┘  │
└──────────────────────┬──────────────────────────────────┘
                       │
        ┌──────────────┼──────────────┐
        │              │              │
        ▼              ▼              ▼
┌───────────────┐ ┌───────────────┐ ┌───────────────┐
│ Member        │ │ Member        │ │ Member        │
│ Cluster 1     │ │ Cluster 2     │ │ Cluster 3     │
│               │ │               │ │               │
│ ┌───────────┐ │ │ ┌───────────┐ │ │ ┌───────────┐ │
│ │ Workloads │ │ │ │ Workloads │ │ │ │ Workloads │ │
│ └───────────┘ │ │ └───────────┘ │ │ └───────────┘ │
└───────────────┘ └───────────────┘ └───────────────┘
```

**KubeFed的优势：**

1. **统一管理**：从单一控制平面管理多个集群
2. **声明式API**：使用Kubernetes原生API风格
3. **灵活调度**：支持复杂的调度策略
4. **高可用**：自动故障转移和负载均衡
5. **可扩展**：支持自定义资源类型

#### KubeFed vs 其他方案

| 特性 | KubeFed | Rancher | ArgoCD |
|------|---------|---------|--------|
| **管理方式** | 声明式API | Web UI + API | GitOps |
| **学习曲线** | 中等 | 低 | 中等 |
| **扩展性** | 高 | 中 | 高 |
| **社区支持** | CNCF项目 | 商业支持 | CNCF项目 |
| **适用场景** | 多集群编排 | 全栈管理 | CI/CD |

### 16.2.2 KubeFed架构深入

#### 核心组件

**1. Federation Controller Manager**

负责管理联邦资源的生命周期：

```go
// Federation Controller核心逻辑
type FederationController struct {
    // 联邦资源定义
    federatedTypeConfig *fedv1b1.FederatedTypeConfig

    // 成员集群客户端
    clusterClients map[string]kubeclient.Interface

    // 调度器
    scheduler *Scheduler

    // 同步控制器
    syncController *SyncController
}

// 处理联邦资源
func (fc *FederationController) Reconcile(fedResource *unstructured.Unstructured) error {
    // 1. 解析放置策略
    placement := fc.parsePlacement(fedResource)

    // 2. 调度到目标集群
    targetClusters := fc.scheduler.Schedule(fedResource, placement)

    // 3. 同步到各集群
    for _, cluster := range targetClusters {
        if err := fc.syncController.SyncToCluster(cluster, fedResource); err != nil {
            return err
        }
    }

    return nil
}
```

**2. Scheduler Controller**

负责决定资源应该部署到哪些集群：

```yaml
# 调度策略示例
apiVersion: core.kubefed.io/v1beta1
kind: FederatedDeployment
metadata:
  name: my-app
  namespace: default
spec:
  # 模板定义
  template:
    metadata:
      labels:
        app: my-app
    spec:
      replicas: 3
      selector:
        matchLabels:
          app: my-app
      template:
        metadata:
          labels:
            app: my-app
        spec:
          containers:
          - name: app
            image: my-app:v1.0

  # 放置策略
  placement:
    clusters:
    - name: cluster-1
    - name: cluster-2
    clusterSelector:
      matchLabels:
        environment: production

  # 覆盖策略
  overrides:
  - clusterName: cluster-1
    clusterOverrides:
    - path: "/spec/replicas"
      value: 5
  - clusterName: cluster-2
    clusterOverrides:
    - path: "/spec/replicas"
      value: 3
```

**3. Sync Controller**

负责将资源同步到成员集群：

```
同步流程：
┌─────────────────────────────────────────────────────┐
│  1. 监听联邦资源变化                                 │
│     ↓                                                │
│  2. 对比期望状态与实际状态                           │
│     ↓                                                │
│  3. 生成同步操作                                     │
│     ↓                                                │
│  4. 应用到目标集群                                   │
│     ↓                                                │
│  5. 更新状态                                         │
└─────────────────────────────────────────────────────┘
```

### 16.2.6 高级调度策略

#### ReplicaSchedulingPreference

ReplicaSchedulingPreference (RSP) 允许你定义如何在多个集群之间分配副本。

**基本概念：**

```
副本分配策略：
┌─────────────────────────────────────────────────────────┐
│  ReplicaSchedulingPreference                            │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  总副本数: 10                                            │
│                                                         │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐   │
│  │ Cluster 1   │  │ Cluster 2   │  │ Cluster 3   │   │
│  │ 权重: 3     │  │ 权重: 2     │  │ 权重: 1     │   │
│  │ 副本: 5     │  │ 副本: 3     │  │ 副本: 2     │   │
│  └─────────────┘  └─────────────┘  └─────────────┘   │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**策略类型：**

**1. 加权分配 (Weighted)**

```yaml
# rsp-weighted.yaml
apiVersion: scheduling.kubefed.io/v1alpha1
kind: ReplicaSchedulingPreference
metadata:
  name: nginx-rsp
  namespace: default
spec:
  targetKind: FederatedDeployment
  totalReplicas: 10

  # 加权分配
  clusters:
    cluster-1:
      weight: 3
    cluster-2:
      weight: 2
    cluster-3:
      weight: 1

  # 重新平衡
  rebalance: true
```

**计算逻辑：**

```
总权重 = 3 + 2 + 1 = 6
cluster-1: 10 * (3/6) = 5 副本
cluster-2: 10 * (2/6) = 3 副本
cluster-3: 10 * (1/6) = 2 副本
```

**2. 最小副本数 (MinReplicas)**

```yaml
# rsp-min-replicas.yaml
apiVersion: scheduling.kubefed.io/v1alpha1
kind: ReplicaSchedulingPreference
metadata:
  name: nginx-rsp
  namespace: default
spec:
  targetKind: FederatedDeployment
  totalReplicas: 10

  clusters:
    cluster-1:
      minReplicas: 3
      weight: 2
    cluster-2:
      minReplicas: 2
      weight: 2
    cluster-3:
      minReplicas: 1
      weight: 1

  rebalance: true
```

**分配逻辑：**

```
1. 首先满足最小副本数：
   cluster-1: 3 (min)
   cluster-2: 2 (min)
   cluster-3: 1 (min)
   已分配: 6

2. 剩余副本按权重分配：
   剩余: 10 - 6 = 4
   总权重: 2 + 2 + 1 = 5
   cluster-1: 3 + 4*(2/5) = 3 + 2 = 5
   cluster-2: 2 + 4*(2/5) = 2 + 2 = 4
   cluster-3: 1 + 4*(1/5) = 1 + 1 = 2
```

**3. 最大副本数 (MaxReplicas)**

```yaml
# rsp-max-replicas.yaml
apiVersion: scheduling.kubefed.io/v1alpha1
kind: ReplicaSchedulingPreference
metadata:
  name: nginx-rsp
  namespace: default
spec:
  targetKind: FederatedDeployment
  totalReplicas: 20

  clusters:
    cluster-1:
      minReplicas: 2
      maxReplicas: 8
      weight: 3
    cluster-2:
      minReplicas: 2
      maxReplicas: 6
      weight: 2
    cluster-3:
      minReplicas: 1
      maxReplicas: 4
      weight: 1

  rebalance: true
```

**4. 动态重新平衡**

```yaml
# rsp-rebalance.yaml
apiVersion: scheduling.kubefed.io/v1alpha1
kind: ReplicaSchedulingPreference
metadata:
  name: nginx-rsp
  namespace: default
spec:
  targetKind: FederatedDeployment
  totalReplicas: 10

  clusters:
    cluster-1:
      weight: 1
    cluster-2:
      weight: 1
    cluster-3:
      weight: 1

  # 启用重新平衡
  rebalance: true

  # 重新平衡触发条件
  rebalanceOnClusterChange: true
```

**重新平衡场景：**

```
场景1：集群故障
初始状态：
  cluster-1: 3 副本 ✅
  cluster-2: 3 副本 ✅
  cluster-3: 4 副本 ✅

cluster-2故障后：
  cluster-1: 5 副本 ✅ (增加2个)
  cluster-2: 0 副本 ❌ (故障)
  cluster-3: 5 副本 ✅ (增加1个)

场景2：新增集群
初始状态：
  cluster-1: 5 副本
  cluster-2: 5 副本

新增cluster-3后：
  cluster-1: 3 副本 (减少2个)
  cluster-2: 3 副本 (减少2个)
  cluster-3: 4 副本 (新增)
```

**5. 基于资源的调度**

```yaml
# rsp-resource-based.yaml
apiVersion: scheduling.kubefed.io/v1alpha1
kind: ReplicaSchedulingPreference
metadata:
  name: nginx-rsp
  namespace: default
spec:
  targetKind: FederatedDeployment
  totalReplicas: 10

  # 基于集群可用资源动态分配
  clusters:
    cluster-1:
      minReplicas: 2
      weight: 1
    cluster-2:
      minReplicas: 2
      weight: 1
    cluster-3:
      minReplicas: 1
      weight: 1

  # 启用资源感知调度
  resourceAware: true

  # 资源阈值
  resourceThresholds:
    cpu: 80%
    memory: 80%
```

**应用RSP：**

```bash
# 创建RSP
kubectl apply -f rsp-weighted.yaml

# 创建FederatedDeployment（不指定replicas）
cat <<EOF | kubectl apply -f -
apiVersion: types.kubefed.io/v1beta1
kind: FederatedDeployment
metadata:
  name: nginx-deployment
  namespace: default
spec:
  template:
    metadata:
      labels:
        app: nginx
    spec:
      replicas: 10  # 将被RSP覆盖
      selector:
        matchLabels:
          app: nginx
      template:
        metadata:
          labels:
            app: nginx
        spec:
          containers:
          - name: nginx
            image: nginx:1.21

  placement:
    clusters:
    - name: cluster-1
    - name: cluster-2
    - name: cluster-3
EOF

# 查看副本分配
kubectl get federateddeployment nginx-deployment -o yaml

# 验证各集群的副本数
kubectl --context cluster-1 get deployment nginx-deployment
kubectl --context cluster-2 get deployment nginx-deployment
kubectl --context cluster-3 get deployment nginx-deployment
```

```yaml
      interval: 30s
      rules:
      # 集群不健康告警
      - alert: KubeFedClusterUnhealthy
        expr: kubefed_cluster_health_status == 0
        for: 5m
        labels:
          severity: critical
        annotations:
          summary: "KubeFed cluster {{ $labels.cluster }} is unhealthy"
          description: "Cluster {{ $labels.cluster }} has been unhealthy for more than 5 minutes."

      # 同步失败告警
      - alert: KubeFedSyncFailure
        expr: rate(kubefed_sync_status{status="failure"}[5m]) > 0.1
        for: 10m
        labels:
          severity: warning
        annotations:
          summary: "High sync failure rate in cluster {{ $labels.cluster }}"
          description: "Sync failure rate is {{ $value }} per second in cluster {{ $labels.cluster }}."

      # 同步延迟告警
      - alert: KubeFedHighSyncLatency
        expr: histogram_quantile(0.95, rate(kubefed_sync_duration_seconds_bucket[5m])) > 10
        for: 15m
        labels:
          severity: warning
        annotations:
          summary: "High sync latency detected"
          description: "95th percentile sync duration is {{ $value }}s, which exceeds 10s threshold."

      # 控制器队列积压告警
      - alert: KubeFedControllerQueueBacklog
        expr: kubefed_controller_queue_depth > 100
        for: 10m
        labels:
          severity: warning
        annotations:
          summary: "KubeFed controller queue backlog"
          description: "Controller {{ $labels.controller }} queue depth is {{ $value }}, indicating processing delays."

      # 副本分配不均告警
      - alert: KubeFedReplicaImbalance
        expr: |
          (max(kubefed_replica_distribution) by (deployment) -
           min(kubefed_replica_distribution) by (deployment)) > 5
        for: 30m
        labels:
          severity: info
        annotations:
          summary: "Replica distribution imbalance for {{ $labels.deployment }}"
          description: "Replica count difference between clusters exceeds 5."
```

## 16.3 Cluster API

### 16.3.1 Cluster API概述

Cluster API (CAPI) 是Kubernetes的一个子项目，提供声明式API和工具来简化跨多个基础设施提供商的Kubernetes集群的配置、升级和操作。

#### 核心价值

**传统集群管理的挑战：**

```
传统方式的问题：
┌─────────────────────────────────────────────────────────┐
│  手动操作                                                │
├─────────────────────────────────────────────────────────┤
│  ❌ 不同云平台使用不同工具                                │
│  ❌ 手动执行升级步骤                                      │
│  ❌ 难以标准化和自动化                                    │
│  ❌ 缺乏统一的API                                        │
│  ❌ 配置漂移风险                                         │
└─────────────────────────────────────────────────────────┘

Cluster API的解决方案：
┌─────────────────────────────────────────────────────────┐
│  声明式管理                                              │
├─────────────────────────────────────────────────────────┤
│  ✅ 统一的Kubernetes API                                │
│  ✅ 自动化生命周期管理                                    │
│  ✅ 多云支持                                             │
│  ✅ GitOps友好                                          │
│  ✅ 可扩展架构                                           │
└─────────────────────────────────────────────────────────┘
```

**主要特性：**

1. **声明式API** - 使用Kubernetes资源定义集群
2. **多云支持** - 统一管理AWS、Azure、GCP、vSphere等
3. **自动化** - 自动处理集群创建、升级、扩缩容
4. **可扩展** - 通过Provider模式支持新的基础设施
5. **GitOps** - 与GitOps工具无缝集成

#### 核心概念

**管理集群与工作负载集群：**

```
Cluster API架构：
┌─────────────────────────────────────────────────────────┐
│  Management Cluster (管理集群)                           │
```

#### 集群生命周期

**创建流程：**

```
集群创建流程：
1. 用户创建Cluster资源
   │
   ▼
2. Cluster Controller创建基础设施
   ├─> 创建InfrastructureCluster (如AWSCluster)
   ├─> 创建网络资源 (VPC, Subnet, etc.)
   └─> 创建负载均衡器
   │
   ▼
3. Control Plane Controller创建控制平面
   ├─> 创建第一个控制平面节点
   ├─> 初始化Kubernetes
   ├─> 生成证书和kubeconfig
   └─> 创建其他控制平面节点
   │
   ▼
4. Machine Controller创建工作节点
   ├─> 创建InfrastructureMachine (如AWSMachine)
   ├─> 创建虚拟机实例
   ├─> 运行bootstrap脚本
   └─> 节点加入集群
   │
   ▼
5. 集群就绪
   └─> 状态更新为Ready
```

**删除流程：**

```
集群删除流程：
1. 用户删除Cluster资源
   │
   ▼
2. Machine Controller删除工作节点
   ├─> 驱逐Pod
   ├─> 删除虚拟机
   └─> 清理资源
   │
   ▼
3. Control Plane Controller删除控制平面
   ├─> 删除控制平面节点
   └─> 清理etcd数据
   │
   ▼
4. Infrastructure Controller清理基础设施
   ├─> 删除负载均衡器
   ├─> 删除网络资源
   └─> 删除其他云资源
   │
   ▼
5. 清理完成
   └─> Cluster资源被删除
```

### 16.3.8 ClusterClass (实验性)

ClusterClass提供了集群模板化能力，允许从单一定义创建多个相似的集群。

**定义ClusterClass：**

```yaml
# cluster-class.yaml
apiVersion: cluster.x-k8s.io/v1beta1
kind: ClusterClass
metadata:
  name: aws-production-class
  namespace: default
spec:
  # 控制平面定义
  controlPlane:
    ref:
      apiVersion: controlplane.cluster.x-k8s.io/v1beta1
      kind: KubeadmControlPlaneTemplate
      name: aws-production-control-plane

    machineInfrastructure:
      ref:
        apiVersion: infrastructure.cluster.x-k8s.io/v1beta2
        kind: AWSMachineTemplate
        name: aws-production-control-plane-machine

  # 基础设施定义
  infrastructure:
    ref:
      apiVersion: infrastructure.cluster.x-k8s.io/v1beta2
      kind: AWSClusterTemplate
      name: aws-production-cluster

  # 工作节点定义
  workers:
    machineDeployments:
    - class: default-worker
      template:
        bootstrap:
          ref:
            apiVersion: bootstrap.cluster.x-k8s.io/v1beta1
            kind: KubeadmConfigTemplate
            name: aws-production-worker-bootstrap
        infrastructure:
          ref:
            apiVersion: infrastructure.cluster.x-k8s.io/v1beta2
            kind: AWSMachineTemplate
            name: aws-production-worker-machine

  # 变量定义
  variables:
  - name: region
    required: true
    schema:
      openAPIV3Schema:
        type: string
        default: us-west-2
  - name: sshKeyName
    required: true
    schema:
      openAPIV3Schema:
        type: string
  - name: controlPlaneInstanceType
    required: false
    schema:
      openAPIV3Schema:
        type: string
        default: t3.large
  - name: workerInstanceType
    required: false
    schema:
      openAPIV3Schema:
        type: string
        default: t3.medium

  # 补丁策略
  patches:
  - name: region
    definitions:
    - selector:
        apiVersion: infrastructure.cluster.x-k8s.io/v1beta2
        kind: AWSClusterTemplate
        matchResources:
          infrastructureCluster: true
      jsonPatches:
      - op: add
        path: /spec/template/spec/region
        valueFrom:
          variable: region

  - name: sshKeyName
    definitions:
    - selector:
        apiVersion: infrastructure.cluster.x-k8s.io/v1beta2
        kind: AWSMachineTemplate
      jsonPatches:
      - op: add
        path: /spec/template/spec/sshKeyName
        valueFrom:
          variable: sshKeyName
```

```bash
    status: "False"
```

## 16.4 多集群服务网格

### 16.4.1 多集群服务网格概述

服务网格在多集群环境下提供统一的服务通信、安全和可观测性能力，是构建分布式微服务架构的关键技术。

**单集群服务网格的局限：**

5. **可观测性** - 统一的指标、日志、追踪

**配置地理位置感知路由：**

```yaml
# locality-routing.yaml
apiVersion: networking.istio.io/v1beta1
kind: DestinationRule
metadata:
  name: helloworld-locality
  namespace: sample
spec:
  host: helloworld.sample.svc.cluster.local

  trafficPolicy:
    # 连接池配置
    connectionPool:
      tcp:
        maxConnections: 100
      http:
        http1MaxPendingRequests: 100
        http2MaxRequests: 100

    # 负载均衡配置
    loadBalancer:
      localityLbSetting:
        enabled: true
        # 故障转移配置
        failover:
        - from: us-west/zone1
          to: us-west/zone2
        - from: us-west/zone2
          to: us-east/zone1

        # 流量分布
        distribute:
        - from: us-west/*
          to:
            "us-west/zone1": 70
            "us-west/zone2": 30
        - from: us-east/*
          to:
            "us-east/zone1": 80
            "us-east/zone2": 20

    # 异常检测
    outlierDetection:
      consecutiveErrors: 5
      interval: 30s
      baseEjectionTime: 30s
      maxEjectionPercent: 50
      minHealthPercent: 40
```

```yaml
# authorization-policy.yaml
apiVersion: security.istio.io/v1beta1
kind: AuthorizationPolicy
metadata:
  name: helloworld-authz
  namespace: sample
spec:
  selector:
    matchLabels:
      app: helloworld

  action: ALLOW

  rules:
  # 允许来自sleep服务的请求
  - from:
    - source:
        principals:
        - "cluster.local/ns/sample/sa/sleep"
    to:
    - operation:

    - source:
        namespaces: ["sample"]
    to:
    - operation:
        methods: ["GET"]

    - key: request.headers[x-cluster-name]
      values: ["cluster1", "cluster2"]

---

apiVersion: security.istio.io/v1beta1

  namespace: sample
spec:
  selector:
    matchLabels:
      app: helloworld

  action: DENY

  rules:
  # 拒绝来自外部的直接访问
```

## 16.5 跨集群负载均衡

### 16.5.1 跨集群负载均衡概述

跨集群负载均衡是多集群架构中实现高可用和流量分发的关键技术，确保用户请求能够智能地路由到最优的集群。

**单集群负载均衡的局限：**

**实现自动故障转移脚本：**

```bash
#!/bin/bash
# auto-failover.sh

# 配置
PRIMARY_CLUSTER="cluster1"
SECONDARY_CLUSTER="cluster2"
HEALTH_CHECK_URL="https://api.example.com/healthz"
CHECK_INTERVAL=10
FAILURE_THRESHOLD=3

# 状态变量
consecutive_failures=0
current_active="$PRIMARY_CLUSTER"

echo "Starting auto-failover monitor..."

while true; do
  # 健康检查
  if curl -sf --max-time 5 "$HEALTH_CHECK_URL" > /dev/null 2>&1; then
    echo "$(date): Health check passed"
    consecutive_failures=0

    # 如果当前是备用集群，尝试切回主集群
    if [ "$current_active" = "$SECONDARY_CLUSTER" ]; then
      echo "$(date): Attempting to failback to primary cluster..."
      # 检查主集群健康
      if check_cluster_health "$PRIMARY_CLUSTER"; then
        failback_to_primary
      fi
    fi
  else
    echo "$(date): Health check failed"
    ((consecutive_failures++))

    # 达到阈值，执行故障转移
    if [ $consecutive_failures -ge $FAILURE_THRESHOLD ]; then
      echo "$(date): Failure threshold reached, initiating failover..."

      if [ "$current_active" = "$PRIMARY_CLUSTER" ]; then
        failover_to_secondary
      else
        echo "$(date): Already on secondary cluster, alerting..."
        send_alert "Both clusters unhealthy!"
      fi

      consecutive_failures=0
    fi
  fi

  sleep $CHECK_INTERVAL
done

function check_cluster_health() {
  local cluster=$1
  # 实现集群健康检查逻辑
  kubectl --context="$cluster" get nodes > /dev/null 2>&1
  return $?
}

function failover_to_secondary() {
  echo "$(date): Failing over to secondary cluster..."

  # 更新DNS权重
  aws route53 change-resource-record-sets \
    --hosted-zone-id <zone-id> \
    --change-batch '{
      "Changes": [
        {
          "Action": "UPSERT",
          "ResourceRecordSet": {
            "Name": "api.example.com",
            "Type": "A",
            "SetIdentifier": "Primary",
            "Weight": 0,
            "TTL": 60,
            "ResourceRecords": [{"Value": "<cluster1-ip>"}]
          }
        },
        {
          "Action": "UPSERT",
          "ResourceRecordSet": {
            "Name": "api.example.com",
            "Type": "A",
            "SetIdentifier": "Secondary",
            "Weight": 100,
            "TTL": 60,
            "ResourceRecords": [{"Value": "<cluster2-ip>"}]
          }
        }
      ]
    }'

  current_active="$SECONDARY_CLUSTER"
  send_alert "Failover completed to secondary cluster"
}

function failback_to_primary() {
  echo "$(date): Failing back to primary cluster..."

  # 逐步切回主集群
  for weight in 20 40 60 80 100; do
    echo "$(date): Setting primary weight to $weight%"

    aws route53 change-resource-record-sets \
      --hosted-zone-id <zone-id> \
      --change-batch '{
        "Changes": [
          {
            "Action": "UPSERT",
            "ResourceRecordSet": {
              "Name": "api.example.com",
              "Type": "A",
              "SetIdentifier": "Primary",
              "Weight": '$weight',
              "TTL": 60,
              "ResourceRecords": [{"Value": "<cluster1-ip>"}]
            }
          },
          {
            "Action": "UPSERT",
            "ResourceRecordSet": {
              "Name": "api.example.com",
              "Type": "A",
              "SetIdentifier": "Secondary",
              "Weight": '$((100-weight))',
              "TTL": 60,
              "ResourceRecords": [{"Value": "<cluster2-ip>"}]
            }
          }
        ]
      }'

    sleep 60
  done

  current_active="$PRIMARY_CLUSTER"
  send_alert "Failback completed to primary cluster"
}

function send_alert() {
  local message=$1
  echo "$(date): ALERT - $message"

  # 发送Slack通知
  curl -X POST https://hooks.slack.com/services/xxx \
    -H 'Content-Type: application/json' \
    -d '{
      "text": "'"$message"'",
      "username": "Failover Monitor",
      "icon_emoji": ":warning:"
    }'

  # 发送邮件
  echo "$message" | mail -s "Cluster Failover Alert" ops@example.com
}
```

```bash
# - 地理距离远
# - 网络路径不优
# - 带宽限制
# - 路由配置不当

# 1. 启用地理就近路由
# 2. 使用Anycast (如AWS Global Accelerator)
# 3. 优化应用架构（缓存、CDN）
# 4. 考虑数据本地化
```

```bash
#!/bin/bash
# scripts/validate-deployment.sh

CLUSTER=$1
THRESHOLD_ERROR_RATE=0.01
THRESHOLD_LATENCY_P99=1000

echo "Validating deployment in cluster: $CLUSTER"

# 查询错误率
ERROR_RATE=$(curl -s "http://prometheus.monitoring.svc.cluster.local:9090/api/v1/query?query=sum(rate(istio_requests_total{response_code=~\"5..\",cluster=\"$CLUSTER\"}[5m]))/sum(rate(istio_requests_total{cluster=\"$CLUSTER\"}[5m]))" | jq -r '.data.result[0].value[1]')

if (( $(echo "$ERROR_RATE > $THRESHOLD_ERROR_RATE" | bc -l) )); then
  echo "ERROR: Error rate $ERROR_RATE exceeds threshold $THRESHOLD_ERROR_RATE"
  exit 1
fi

# 查询延迟
LATENCY_P99=$(curl -s "http://prometheus.monitoring.svc.cluster.local:9090/api/v1/query?query=histogram_quantile(0.99,sum(rate(istio_request_duration_milliseconds_bucket{cluster=\"$CLUSTER\"}[5m]))by(le))" | jq -r '.data.result[0].value[1]')

if (( $(echo "$LATENCY_P99 > $THRESHOLD_LATENCY_P99" | bc -l) )); then
  echo "ERROR: P99 latency $LATENCY_P99 ms exceeds threshold $THRESHOLD_LATENCY_P99 ms"
  exit 1
fi

echo "Deployment validation passed"
echo "Error rate: $ERROR_RATE"
echo "P99 latency: $LATENCY_P99 ms"
```
