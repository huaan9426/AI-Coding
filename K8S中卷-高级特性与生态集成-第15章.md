## 15.1 调度器架构深入

### 15.1.1 调度器概述

**什么是Kubernetes调度器？**

Kubernetes调度器（kube-scheduler）是控制平面的核心组件，负责为新创建的Pod选择合适的节点（Node）运行。调度决策考虑了资源需求、硬件/软件/策略约束、亲和性和反亲和性规则、数据本地性、工作负载干扰等因素。

**调度器的核心职责**：

```
调度器的三大职责：
┌────────────────────────────────────────────┐
│  1. 监听（Watch）                          │
│     - 监听API Server的Pod创建事件          │
│     - 过滤出未调度的Pod（.spec.nodeName为空）│
│     - 加入调度队列                         │
└────────────────────────────────────────────┘
                ↓
┌────────────────────────────────────────────┐
│  2. 调度（Schedule）                       │
│     - 预选（Predicates/Filter）            │
│       找出所有满足条件的节点               │
│     - 优选（Priorities/Score）             │
│       给节点打分，选择最优节点             │
│     - 绑定前检查（Reserve）                │
│       预留资源，执行插件                   │
└────────────────────────────────────────────┘
                ↓
┌────────────────────────────────────────────┐
│  3. 绑定（Bind）                           │
│     - 将Pod绑定到选定的节点                │
│     - 更新API Server中Pod的.spec.nodeName  │
│     - 通知相关组件（kubelet）              │
└────────────────────────────────────────────┘
```

**1. Informer机制**

```go
// Informer机制
type Scheduler struct {
```

**2. 调度队列**

```go
// 调度队列的三个子队列
type SchedulingQueue interface {
    // ActiveQ: 活跃队列
    // - 存储待调度的Pod
    // - 按优先级排序（高优先级先调度）
    // - 使用堆（Heap）数据结构
    Pop() (*framework.QueuedPodInfo, error)

    // BackoffQ: 退避队列
    // - 存储调度失败的Pod
    // - 实现指数退避（Exponential Backoff）
    // - 重试间隔：1s, 2s, 4s, 8s, ... 最大64s
    AddUnschedulableIfNotPresent(*framework.QueuedPodInfo, int64) error

    // UnschedulableQ: 不可调度队列
    // - 存储暂时无法调度的Pod
    // - 等待集群状态变化（Node增加、Pod删除等）
    // - 定期将Pod移回ActiveQ重试
    MoveAllToActiveOrBackoffQueue(event string)
}

// 队列流程
┌────────────┐
│  新建Pod   │
└────────────┘
      ↓
┌────────────┐
│  ActiveQ   │ ← ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ┐
└────────────┘                         │
      ↓                                │
  调度尝试                             │
      ↓                                │
  成功？                               │
   ↙  ↘                                │
 Yes   No                              │
  ↓     ↓                              │
 绑定  ┌────────────┐                  │
      │  BackoffQ  │ ─ ─ ─ ─(指数退避)┘
      └────────────┘
           ↓
      (多次失败)
           ↓
      ┌────────────────┐
      │ UnschedulableQ │ ─ ─(集群变化)─ ┘
      └────────────────┘
```

**3. 调度框架（Scheduling Framework）**

调度框架是Kubernetes 1.15引入的新架构，将调度过程分为多个扩展点（Extension Points）。

```
调度框架的扩展点：
┌─────────────────────────────────────────────────────┐
│                 调度周期                             │
│                                                      │
│  ┌──────────────────────────────────────────────┐  │
│  │  1. PreFilter（预过滤）                      │  │
│  │     - 预处理Pod信息                          │  │
│  │     - 计算前置条件                           │  │
│  │     示例：计算Pod的亲和性要求                │  │
│  └──────────────────────────────────────────────┘  │
│                    ↓                                 │
│  ┌──────────────────────────────────────────────┐  │
│  │  2. Filter（过滤/预选）                      │  │
│  │     - 找出所有可以运行Pod的节点              │  │
│  │     - 并发执行，检查每个节点                 │  │
│  │     示例：检查节点资源是否足够               │  │
│  └──────────────────────────────────────────────┘  │
│                    ↓                                 │
│  ┌──────────────────────────────────────────────┐  │
│  │  3. PostFilter（后过滤）                     │  │
│  │     - 当Filter阶段没有合适节点时触发         │  │
│  │     - 尝试抢占低优先级Pod                    │  │
│  │     示例：抢占逻辑                           │  │
│  └──────────────────────────────────────────────┘  │
│                    ↓                                 │
│  ┌──────────────────────────────────────────────┐  │
│  │  4. PreScore（预打分）                       │  │
│  │     - 预处理打分信息                         │  │
│  │     - 生成可共享的状态                       │  │
│  └──────────────────────────────────────────────┘  │
│                    ↓                                 │
│  ┌──────────────────────────────────────────────┐  │
│  │  5. Score（打分/优选）                       │  │
│  │     - 给每个节点打分                         │  │
│  │     - 并发执行                               │  │
│  │     示例：资源利用率打分                     │  │
│  └──────────────────────────────────────────────┘  │
│                    ↓                                 │
│  ┌──────────────────────────────────────────────┐  │
│  │  6. NormalizeScore（归一化打分）             │  │
│  │     - 将不同插件的分数归一化到0-100          │  │
│  │     - 应用插件权重                           │  │
│  └──────────────────────────────────────────────┘  │
│                    ↓                                 │
│  ┌──────────────────────────────────────────────┐  │
│  │  7. Reserve（预留）                          │  │
│  │     - 维护运行时状态                         │  │
│  │     - 预留资源                               │  │
│  │     示例：预留GPU、RDMA等资源                │  │
│  └──────────────────────────────────────────────┘  │
│                    ↓                                 │
│  ┌──────────────────────────────────────────────┐  │
│  │  8. Permit（许可）                           │  │
│  │     - 批准或拒绝调度结果                     │  │
│  │     - 可以等待外部批准                       │  │
│  │     示例：等待资源配额检查                   │  │
│  └──────────────────────────────────────────────┘  │
│                                                      │
└─────────────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────────────┐
│                 绑定周期（异步）                     │
│                                                      │
│  ┌──────────────────────────────────────────────┐  │
│  │  9. PreBind（预绑定）                        │  │
│  │     - 绑定前的准备工作                       │  │
│  │     - 如挂载网络卷                           │  │
│  │     示例：预挂载PV                           │  │
│  └──────────────────────────────────────────────┘  │
│                    ↓                                 │
│  ┌──────────────────────────────────────────────┐  │
│  │  10. Bind（绑定）                            │  │
│  │      - 将Pod绑定到节点                       │  │
│  │      - 更新API Server                        │  │
│  └──────────────────────────────────────────────┘  │
│                    ↓                                 │
│  ┌──────────────────────────────────────────────┐  │
│  │  11. PostBind（后绑定）                      │  │
│  │      - 绑定后的清理工作                      │  │
│  │      - 发送事件通知                          │  │
│  └──────────────────────────────────────────────┘  │
│                                                      │
└─────────────────────────────────────────────────────┘
```

### 15.1.3 调度流程详解

**完整的调度流程**：

```
详细调度流程：
┌───────────────────────────────────────────────────┐
│  Step 1: 监听Pod创建事件                          │
│  ────────────────────────────────────────────     │
│  Informer监听API Server                           │
│  ↓                                                 │
│  检测到新创建的Pod（spec.nodeName为空）           │
│  ↓                                                 │
│  将Pod加入ActiveQ（按优先级排序）                 │
└───────────────────────────────────────────────────┘
                    ↓
┌───────────────────────────────────────────────────┐
│  Step 2: 从队列中取出Pod                          │
│  ────────────────────────────────────────────     │
│  从ActiveQ Pop一个最高优先级的Pod                 │
│  ↓                                                 │
│  检查Pod是否可以调度                              │
│  - 是否被删除                                     │
│  - 是否已经调度                                   │
│  - 是否在调度中                                   │
└───────────────────────────────────────────────────┘
                    ↓
┌───────────────────────────────────────────────────┐
│  Step 3: PreFilter阶段                            │
│  ────────────────────────────────────────────     │
│  执行PreFilter插件：                              │
│  ├─ NodePorts: 检查Pod需要的端口                  │
│  ├─ NodeAffinity: 解析节点亲和性要求              │
│  ├─ PodTopologySpread: 计算拓扑分布约束           │
│  └─ InterPodAffinity: 计算Pod间亲和性             │
│  ↓                                                 │
│  生成PreFilter状态（供后续插件使用）              │
└───────────────────────────────────────────────────┘
                    ↓
┌───────────────────────────────────────────────────┐
│  Step 4: Filter阶段（预选）                       │
│  ────────────────────────────────────────────     │
│  并发检查每个节点：                               │
│  ┌────────────────────────────────────────────┐  │
│  │ Node 1                                     │  │
│  │ ├─ NodeUnschedulable: 节点是否可调度      │  │
│  │ ├─ NodeResourcesFit: 资源是否足够         │  │
│  │ ├─ NodeName: 是否指定了节点名             │  │
│  │ ├─ NodePorts: 端口是否冲突                │  │
│  │ ├─ NodeAffinity: 节点亲和性是否满足       │  │
│  │ ├─ VolumeBinding: 存储卷是否可用          │  │
│  │ ├─ TaintToleration: 污点容忍是否匹配      │  │
│  │ ├─ PodTopologySpread: 拓扑分布是否满足    │  │
│  │ └─ InterPodAffinity: Pod亲和性是否满足    │  │
│  │ 结果: ✅ 通过                              │  │
│  └────────────────────────────────────────────┘  │
│  ┌────────────────────────────────────────────┐  │
│  │ Node 2                                     │  │
│  │ ├─ ... 检查所有Filter插件                 │  │
│  │ 结果: ❌ 资源不足                          │  │
│  └────────────────────────────────────────────┘  │
│  ┌────────────────────────────────────────────┐  │
│  │ Node 3                                     │  │
│  │ ├─ ... 检查所有Filter插件                 │  │
│  │ 结果: ✅ 通过                              │  │
│  └────────────────────────────────────────────┘  │
│  ↓                                                 │
│  过滤结果: [Node 1, Node 3]                       │
```

### 15.1.4 调度算法原理

**1. Filter算法（预选）**

Filter阶段使用多个插件并发检查每个节点，任何一个插件返回失败，该节点就被过滤掉。

**核心Filter插件**：

```yaml
# 1. NodeResourcesFit（资源适配）
算法: 检查节点的可分配资源是否满足Pod的资源请求

计算公式:
  NodeAllocatable = NodeCapacity - SystemReserved - KubeReserved
  PodRequest = sum(containers[].resources.requests)

  if PodRequest > (NodeAllocatable - NodeAllocated):
      return Fail
  else:
      return Success

示例:
  Node资源:
    - Capacity: CPU=8, Memory=16Gi
    - SystemReserved: CPU=0.5, Memory=1Gi
    - KubeReserved: CPU=0.5, Memory=1Gi
    - Allocatable: CPU=7, Memory=14Gi
    - Allocated: CPU=5, Memory=10Gi
    - Available: CPU=2, Memory=4Gi

  Pod请求:
    - CPU=1, Memory=2Gi

  结果: ✅ Success (1 < 2 and 2Gi < 4Gi)

---

# 2. NodeAffinity（节点亲和性）
算法: 检查Pod的节点亲和性是否匹配节点标签

匹配规则:
  requiredDuringSchedulingIgnoredDuringExecution:
    - 必须满足（硬约束）
    - 不满足则过滤该节点

  preferredDuringSchedulingIgnoredDuringExecution:
    - 尽量满足（软约束）
    - 用于打分阶段

示例:
  Pod亲和性:
    requiredDuringSchedulingIgnoredDuringExecution:
      nodeSelectorTerms:
      - matchExpressions:
        - key: disktype
          operator: In
          values: ["ssd"]

  Node标签:
    disktype: ssd

  结果: ✅ Success

---

# 3. PodTopologySpread（拓扑分布）
算法: 检查Pod在拓扑域的分布是否满足约束

计算公式:
  Skew = PodCountInDomain - MinPodCount

  if Skew > MaxSkew:
      return Fail
  else:
      return Success

示例:
  拓扑约束:
    maxSkew: 1
    topologyKey: zone
    whenUnsatisfiable: DoNotSchedule

  当前分布:
    zone-a: 2 pods
    zone-b: 1 pod
    zone-c: 1 pod

  调度到zone-a:
    Skew = 3 - 1 = 2 > 1
    结果: ❌ Fail

  调度到zone-b:
    Skew = 2 - 1 = 1 <= 1
    结果: ✅ Success

---

# 4. TaintToleration（污点容忍）
算法: 检查Pod的容忍度是否匹配节点的污点

匹配规则:
  对于节点的每个污点:
    - 如果Pod有匹配的容忍度，则忽略该污点
    - 如果没有匹配的容忍度:
      - Effect=NoSchedule: 不可调度
      - Effect=PreferNoSchedule: 尽量不调度（软约束）
      - Effect=NoExecute: 不可调度且驱逐已有Pod

示例:
  Node污点:
    - key: gpu
      value: "true"
      effect: NoSchedule

  Pod容忍度:
    - key: gpu
      operator: Equal
      value: "true"
      effect: NoSchedule

  结果: ✅ Success (容忍度匹配)

---

# 5. VolumeBinding（存储卷绑定）
算法: 检查Pod需要的PV是否可以绑定到该节点

检查项:
  1. PVC是否已绑定到PV
  2. PV的节点亲和性是否匹配当前节点
  3. 动态PV: StorageClass是否支持该节点
  4. 本地PV: 是否在该节点上

示例:
  PVC需求:
    storageClassName: local-storage
    accessMode: ReadWriteOnce

  PV (Node 1):
    nodeAffinity:
      required:
        nodeSelectorTerms:
        - matchExpressions:
          - key: kubernetes.io/hostname
            operator: In
            values: ["node-1"]

  调度到Node 1: ✅ Success
  调度到Node 2: ❌ Fail (PV不在Node 2上)

---

# 6. InterPodAffinity（Pod间亲和性）
算法: 检查Pod间的亲和性和反亲和性

匹配规则:
  requiredDuringSchedulingIgnoredDuringExecution:
    - 亲和性: 必须与匹配的Pod在同一拓扑域
    - 反亲和性: 不能与匹配的Pod在同一拓扑域

  preferredDuringSchedulingIgnoredDuringExecution:
    - 软约束，用于打分

示例:
  Pod反亲和性:
    requiredDuringSchedulingIgnoredDuringExecution:
    - labelSelector:
        matchLabels:
          app: web
      topologyKey: kubernetes.io/hostname

  Node 1上有Pod (app=web)
  Node 2上没有Pod (app=web)

  调度到Node 1: ❌ Fail (违反反亲和性)
  调度到Node 2: ✅ Success
```

**2. Score算法（优选）**

Score阶段给每个候选节点打分，选择得分最高的节点。

**核心Score插件**：

```yaml
# 1. NodeResourcesBalancedAllocation（资源均衡分配）
算法: 节点的CPU和内存使用率越接近，分数越高

计算公式:
  CPUFraction = (PodRequestCPU + NodeAllocatedCPU) / NodeAllocatableCPU
  MemoryFraction = (PodRequestMemory + NodeAllocatedMemory) / NodeAllocatableMemory

  Score = 100 - abs(CPUFraction - MemoryFraction) * 100

示例:
  Node 1:
    CPU: 50% 使用
    Memory: 40% 使用
    差值: |50% - 40%| = 10%
    分数: 100 - 10 = 90

  Node 2:
    CPU: 80% 使用
    Memory: 30% 使用
    差值: |80% - 30%| = 50%
    分数: 100 - 50 = 50

  Node 1得分更高 ✅

---

# 2. NodeResourcesLeastAllocated（最少分配）
算法: 节点空闲资源越多，分数越高

计算公式:
  CPUScore = (NodeAllocatableCPU - NodeAllocatedCPU) / NodeAllocatableCPU * 100
  MemoryScore = (NodeAllocatableMemory - NodeAllocatedMemory) / NodeAllocatableMemory * 100

  Score = (CPUScore + MemoryScore) / 2

示例:
  Node 1:
    CPU空闲: 50%
    Memory空闲: 60%
    分数: (50 + 60) / 2 = 55

  Node 2:
    CPU空闲: 70%
    Memory空闲: 80%
    分数: (70 + 80) / 2 = 75

  Node 2得分更高 ✅

---

# 3. ImageLocality（镜像本地性）
算法: 镜像已在节点上，分数越高（减少拉取时间）

计算公式:
  ImageSize = sum(已在节点的镜像大小)
  MaxImageSize = 1000Mi (配置值)

  Score = min(ImageSize / MaxImageSize * 100, 100)

示例:
  Pod镜像: nginx:1.21 (50Mi), app:1.0 (200Mi)

  Node 1:
    已有镜像: nginx:1.21 (50Mi)
    分数: 50Mi / 1000Mi * 100 = 5

  Node 2:
    已有镜像: nginx:1.21 (50Mi), app:1.0 (200Mi)
    分数: 250Mi / 1000Mi * 100 = 25

  Node 2得分更高 ✅

---

# 4. NodeAffinity（节点亲和性打分）
算法: 节点满足preferredDuringScheduling越多，分数越高

计算公式:
  Score = sum(满足的preferred条件的权重) / sum(所有preferred条件的权重) * 100

示例:
  Pod节点亲和性:
    preferredDuringSchedulingIgnoredDuringExecution:
    - weight: 50
      preference:
        matchExpressions:
        - key: disktype
          operator: In
          values: ["ssd"]
    - weight: 30
      preference:
        matchExpressions:
        - key: zone
          operator: In
          values: ["zone-a"]

  Node 1标签:
    disktype: ssd
    zone: zone-b
    满足: disktype (权重50)
    分数: 50 / 80 * 100 = 62.5

  Node 2标签:
    disktype: ssd
    zone: zone-a
    满足: disktype + zone (权重80)
    分数: 80 / 80 * 100 = 100

  Node 2得分更高 ✅

---

# 5. InterPodAffinity（Pod间亲和性打分）
算法: 根据Pod的preferred亲和性/反亲和性打分

计算公式:
  对于每个preferred规则:
    - 统计满足条件的Pod数量
    - 分数 += 权重 * 满足数量

  最终归一化到0-100

示例:
  Pod亲和性:
    preferredDuringSchedulingIgnoredDuringExecution:
    - weight: 100
      podAffinityTerm:
        labelSelector:
          matchLabels:
            app: cache
        topologyKey: kubernetes.io/hostname

  Node 1: 有2个 app=cache 的Pod
    分数: 100 * 2 = 200 (归一化后100)

  Node 2: 没有 app=cache 的Pod
    分数: 100 * 0 = 0

  Node 1得分更高 ✅

---

# 6. PodTopologySpread（拓扑分布打分）
算法: 使拓扑域的Pod分布更均衡

计算公式:
  Skew = PodCountInDomain - MinPodCount
  Score = (MaxSkew - Skew) / MaxSkew * 100

示例:
  拓扑约束:
    maxSkew: 2
    topologyKey: zone
    whenUnsatisfiable: ScheduleAnyway

  当前分布:
    zone-a: 3 pods (MinPodCount)
    zone-b: 4 pods
    zone-c: 5 pods

  调度到zone-a:
    Skew = 4 - 3 = 1
    分数: (2 - 1) / 2 * 100 = 50

  调度到zone-c:
    Skew = 6 - 3 = 3
    分数: (2 - 3) / 2 * 100 = -50 (取0)

  zone-a得分更高 ✅
```

**插件权重配置**：

```yaml
# 配置插件权重
apiVersion: kubescheduler.config.k8s.io/v1
kind: KubeSchedulerConfiguration
profiles:
- schedulerName: default-scheduler
  plugins:
    score:
      enabled:
      - name: NodeResourcesBalancedAllocation
        weight: 1
      - name: NodeResourcesLeastAllocated
        weight: 1
      - name: ImageLocality
        weight: 5  # 镜像本地性权重更高
      - name: InterPodAffinity
        weight: 2
      - name: NodeAffinity
        weight: 2
      - name: PodTopologySpread
        weight: 2
```

**最终得分计算**：

```
最终得分 =
  NodeResourcesBalancedAllocation * 1 +
  NodeResourcesLeastAllocated * 1 +
  ImageLocality * 5 +
  InterPodAffinity * 2 +
  NodeAffinity * 2 +
  PodTopologySpread * 2

归一化到0-100范围
```

## 15.2 调度框架（Scheduling Framework）

### 15.2.1 调度框架概述

**什么是调度框架**？

Kubernetes Scheduling Framework是在Kubernetes 1.15版本引入的可插拔调度器架构，它将调度器的核心逻辑抽象为一系列**扩展点（Extension Points）**，开发者可以通过实现插件接口来定制调度行为。

**为什么需要调度框架**？

在调度框架之前，定制调度器有两种方式：
1. **Fork调度器代码**：维护成本高，难以跟随K8s版本升级
2. **编写独立调度器**：需要重新实现大量基础逻辑

调度框架通过插件化设计解决了这些问题：
- ✅ **低侵入性**：无需修改核心代码
- ✅ **高复用性**：可复用内置插件
- ✅ **易维护性**：跟随K8s版本自动升级
- ✅ **高灵活性**：可在多个扩展点注入逻辑

**架构图**：

```

┌─────────────────────────────────────────────────────────────┐
```

### 15.2.2 扩展点详解

调度框架定义了**11个扩展点**，按调度阶段分为两个周期：

#### 1. QueueSort（队列排序）

**作用**：定义调度队列中Pod的排序规则

**接口定义**：
```go
type QueueSortPlugin interface {
    Plugin
    Less(*QueuedPodInfo, *QueuedPodInfo) bool
}
```

**实现示例**：
```go
// 优先级排序插件
type PrioritySort struct{}

func (pl *PrioritySort) Less(pInfo1, pInfo2 *framework.QueuedPodInfo) bool {
    p1 := corev1helpers.PodPriority(pInfo1.Pod)
    p2 := corev1helpers.PodPriority(pInfo2.Pod)
    return p1 > p2  // 高优先级在前
}
```

**使用场景**：
- 实现自定义优先级规则
- 基于业务属性排序（如租户级别、SLA等级）
- 实现公平调度（防止低优先级Pod饥饿）

---

#### 2. PreFilter（预过滤）

**作用**：
- 预处理Pod信息
- 提前计算可复用的数据（如Pod亲和性相关Pod列表）
- 快速拒绝明显无法调度的Pod

**接口定义**：
```go
type PreFilterPlugin interface {
    Plugin
    PreFilter(ctx context.Context, state *CycleState, p *v1.Pod) *Status
    PreFilterExtensions() PreFilterExtensions
}

type PreFilterExtensions interface {
    AddPod(ctx context.Context, state *CycleState,
           podToSchedule *v1.Pod, podToAdd *v1.Pod,
           nodeInfo *NodeInfo) *Status
    RemovePod(ctx context.Context, state *CycleState,
              podToSchedule *v1.Pod, podToRemove *v1.Pod,
              nodeInfo *NodeInfo) *Status
}
```

**实现示例**：
```go
// Pod间亲和性预过滤插件
type InterPodAffinity struct{}

func (pl *InterPodAffinity) PreFilter(ctx context.Context,
    state *CycleState, pod *v1.Pod) *Status {

    // 提前计算所有相关的Pod列表
    affinity := pod.Spec.Affinity
    if affinity == nil || affinity.PodAffinity == nil {
        return nil  // 无亲和性要求，直接返回
    }

    // 查找所有匹配亲和性规则的Pod
    allPods, err := pl.sharedLister.Pods().List(labels.Everything())
    if err != nil {
        return framework.NewStatus(framework.Error, err.Error())
    }

    // 构建状态数据供Filter阶段使用
    affinityState := &affinityState{
        matchingPods: filterMatchingPods(allPods, affinity),
    }
    state.Write(affinityStateKey, affinityState)

    return nil
}
```

**使用场景**：
- 提前计算复杂信息（避免在每个节点重复计算）
- 快速拒绝（如Pod请求的资源超过集群总资源）
- 数据预处理（如构建拓扑域映射）

---

#### 3. Filter（过滤）

**作用**：过滤不满足Pod调度要求的节点

**接口定义**：
```go
type FilterPlugin interface {
    Plugin
    Filter(ctx context.Context, state *CycleState,
           pod *v1.Pod, nodeInfo *NodeInfo) *Status
}
```

**实现示例**：
```go
// 自定义GPU过滤插件
type GPUFilter struct{}

func (pl *GPUFilter) Filter(ctx context.Context,
    state *CycleState, pod *v1.Pod, nodeInfo *framework.NodeInfo) *Status {

    // 检查Pod是否需要GPU
    gpuRequest := getGPURequest(pod)
    if gpuRequest == 0 {
        return nil  // 不需要GPU，通过
    }

    // 检查节点GPU资源
    node := nodeInfo.Node()
    gpuCapacity := getGPUCapacity(node)
    gpuAllocated := getGPUAllocated(nodeInfo)

    gpuAvailable := gpuCapacity - gpuAllocated
    if gpuAvailable < gpuRequest {
        return framework.NewStatus(
            framework.Unschedulable,
            fmt.Sprintf("Insufficient GPU: need %d, available %d",
                        gpuRequest, gpuAvailable),
        )
    }

    // 检查GPU型号匹配
    requiredGPUModel := pod.Annotations["gpu-model"]
    if requiredGPUModel != "" {
        nodeGPUModel := node.Labels["gpu-model"]
        if nodeGPUModel != requiredGPUModel {
            return framework.NewStatus(
                framework.Unschedulable,
                fmt.Sprintf("GPU model mismatch: need %s, got %s",
                            requiredGPUModel, nodeGPUModel),
            )
        }
    }

    return nil  // 通过过滤
}
```

**作用**：

**实现示例**：
```go
// 默认抢占插件
type DefaultPreemption struct{}

func (pl *DefaultPreemption) PostFilter(ctx context.Context,
    state *CycleState, pod *v1.Pod,
    filteredNodeStatusMap framework.NodeToStatusMap) (*framework.PostFilterResult, *Status) {

    // 仅对高优先级Pod执行抢占
    if !isPodEligibleForPreemption(pod) {
        return nil, framework.NewStatus(framework.Unschedulable,
            "preemption is not helpful for this pod")
    }

    // 查找可抢占的节点
    candidateNodes := pl.findCandidates(ctx, pod, filteredNodeStatusMap)
    if len(candidateNodes) == 0 {
        return nil, framework.NewStatus(framework.Unschedulable,
            "no nodes available for preemption")
    }

    // 选择最佳抢占节点（驱逐最少的Pod）
    bestNode := selectBestNode(candidateNodes)

    // 执行抢占：删除低优先级Pod
    victimsToEvict := pl.selectVictims(bestNode, pod)
    for _, victim := range victimsToEvict {
        if err := pl.client.CoreV1().Pods(victim.Namespace).
            Delete(ctx, victim.Name, metav1.DeleteOptions{}); err != nil {
            return nil, framework.NewStatus(framework.Error, err.Error())
        }
    }

    // 返回提名节点
    return &framework.PostFilterResult{
        NominatedNodeName: bestNode.Name,
    }, framework.NewStatus(framework.Success)
}
```

**使用场景**：
- 实现自定义抢占策略
- 触发集群自动扩容（与Cluster Autoscaler集成）
- 记录调度失败原因

```go
// 数据本地化打分插件
type DataLocality struct{}

func (pl *DataLocality) Score(ctx context.Context,
    state *CycleState, pod *v1.Pod, nodeName string) (int64, *Status) {

    // 获取Pod需要访问的数据集
    datasetName := pod.Annotations["dataset"]
    if datasetName == "" {
        return 0, nil  // 无数据本地化要求
    }

    // 检查节点是否有数据缓存
    node := pl.nodeInfoMap[nodeName]
    cachedDatasets := node.Labels["cached-datasets"]

    var score int64 = 0
    if strings.Contains(cachedDatasets, datasetName) {
        // 数据在本地，高分
        score = 100
    } else {
        // 计算网络距离分数
        distance := pl.getNetworkDistance(node, datasetName)
        score = max(0, 100 - distance*10)
    }

    return score, nil
}

func (pl *DataLocality) NormalizeScore(ctx context.Context,
    state *CycleState, pod *v1.Pod, scores framework.NodeScoreList) *Status {

    // 归一化到0-100范围
    var highest int64 = 0
    for _, score := range scores {
        if score.Score > highest {
            highest = score.Score
        }
    }

    if highest == 0 {
        return nil
    }

    for i := range scores {
        scores[i].Score = scores[i].Score * framework.MaxNodeScore / highest
    }

    return nil
}
```

#### 8. Permit（许可）

**作用**：
- 最后的守门人，可以批准、拒绝或等待
- 实现批量调度（Gang Scheduling）

**接口定义**：
```go

    Plugin

}

// 返回状态:
// - Success: 立即允许绑定
// - Unschedulable: 拒绝调度
// - Wait: 等待指定时间（用于Gang调度）
```

**实现示例 - Gang调度**：
```go
// Gang调度插件：确保一组Pod同时调度
type Coscheduling struct {
    frameworkHandle framework.Handle
    waitingPods     map[string]*waitingPod  // podGroupID -> waiting pods
}

type waitingPod struct {
    pods          []*v1.Pod
    minAvailable  int
    waitTimeout   time.Duration
}

func (pl *Coscheduling) Permit(ctx context.Context,
    state *CycleState, pod *v1.Pod, nodeName string) (*framework.Status, time.Duration) {

    // 检查是否属于PodGroup
    podGroupName := pod.Labels["pod-group.scheduling.sigs.k8s.io/name"]
    if podGroupName == "" {
        return framework.NewStatus(framework.Success), 0  // 单独调度
    }

    // 获取PodGroup定义
    podGroup, err := pl.getPodGroup(pod.Namespace, podGroupName)
    if err != nil {
        return framework.NewStatus(framework.Error, err.Error()), 0
    }

    minAvailable := podGroup.Spec.MinMember

    // 记录已调度的Pod
    pl.mu.Lock()
    defer pl.mu.Unlock()

    key := fmt.Sprintf("%s/%s", pod.Namespace, podGroupName)
    waiting, exists := pl.waitingPods[key]
    if !exists {
        waiting = &waitingPod{
            pods:         []*v1.Pod{},
            minAvailable: minAvailable,
            waitTimeout:  60 * time.Second,
        }
        pl.waitingPods[key] = waiting
    }

    waiting.pods = append(waiting.pods, pod)

    // 检查是否满足最小数量
    if len(waiting.pods) >= minAvailable {
        // 批准所有等待的Pod
        for _, p := range waiting.pods {
            pl.frameworkHandle.IterateOverWaitingPods(func(wp framework.WaitingPod) {
                if wp.GetPod().UID == p.UID {
                    wp.Allow(pl.Name())  // 解除等待
                }
            })
        }
        delete(pl.waitingPods, key)
        return framework.NewStatus(framework.Success), 0
    }

    // 继续等待
    klog.Infof("Pod %s waiting for gang scheduling: %d/%d",
               pod.Name, len(waiting.pods), minAvailable)

    return framework.NewStatus(framework.Wait), waiting.waitTimeout
}
```

```go
// 预过滤：预计算价格信息
func (co *CostOptimizer) PreFilter(ctx context.Context,
```

```go
// 过滤：排除超出预算的节点
func (co *CostOptimizer) Filter(ctx context.Context,
    state *framework.CycleState, pod *v1.Pod,
    nodeInfo *framework.NodeInfo) *framework.Status {

    // 读取价格信息
    data, err := state.Read(costStateKey)
    if err != nil {
        return nil  // 无价格信息，不过滤
    }

    costState := data.(*CostState)

    // 检查预算限制
    maxBudget := pod.Annotations["scheduler.cost-optimizer/max-budget"]
    if maxBudget == "" {
        return nil  // 无预算限制
    }

    maxPrice, _ := strconv.ParseFloat(maxBudget, 64)
    nodePrice := costState.NodePrices[nodeInfo.Node().Name]

    if nodePrice > maxPrice {
        return framework.NewStatus(framework.Unschedulable,
            fmt.Sprintf("node price $%.2f/hr exceeds budget $%.2f/hr",
                        nodePrice, maxPrice))
    }

    return nil
}
```

**步骤4：实现Score接口**

```go
// 打分：价格越低分数越高
func (co *CostOptimizer) Score(ctx context.Context,
    state *framework.CycleState, pod *v1.Pod,
    nodeName string) (int64, *framework.Status) {

    // 读取价格信息
    data, err := state.Read(costStateKey)
    if err != nil {
        return 0, nil  // 无价格信息，返回中性分数
    }

    costState := data.(*CostState)
    nodePrice := costState.NodePrices[nodeName]

    // 计算价格分数（价格越低分数越高）
    maxPrice := co.getMaxPrice(costState.NodePrices)
    var priceScore int64
    if maxPrice > 0 {
        priceScore = int64((maxPrice - nodePrice) / maxPrice * 100)
    }

    // 获取性能分数（CPU频率、网络带宽等）
    performanceScore := co.getPerformanceScore(nodeName)

    // 加权计算最终分数
    finalScore := (priceScore * int64(co.args.PriceWeight) +
                   performanceScore * int64(co.args.PerformanceWeight)) / 100

    return finalScore, nil
}

func (co *CostOptimizer) ScoreExtensions() framework.ScoreExtensions {
    return co  // 实现NormalizeScore
}

// 归一化
func (co *CostOptimizer) NormalizeScore(ctx context.Context,
    state *framework.CycleState, pod *v1.Pod,
    scores framework.NodeScoreList) *framework.Status {

    var highest int64 = 0
    for _, score := range scores {
        if score.Score > highest {
            highest = score.Score
        }
    }

    if highest == 0 {
        return nil
    }

    for i := range scores {
        scores[i].Score = scores[i].Score * framework.MaxNodeScore / highest
    }

    return nil
}

// 辅助方法
func (co *CostOptimizer) getMaxPrice(prices map[string]float64) float64 {
    var max float64
    for _, price := range prices {
        if price > max {
            max = price
        }
    }
    return max
}

func (co *CostOptimizer) getPerformanceScore(nodeName string) int64 {
    // 从节点标签读取性能指标
    node, _ := co.handle.ClientSet().CoreV1().Nodes().
        Get(context.TODO(), nodeName, metav1.GetOptions{})

    cpuFreq := node.Labels["cpu-frequency"]
    networkBW := node.Labels["network-bandwidth"]

    // 简化的性能计算
    var score int64 = 50  // 默认中等性能

    if cpuFreq == "high" {
        score += 25
    }
    if networkBW == "10Gbps" {
        score += 25
    }

    return score
}

func (co *CostOptimizer) getNodePrices() (map[string]float64, error) {
    // 实际场景中从云服务商API或ConfigMap读取
    // 这里使用节点标签模拟
    nodes, err := co.handle.ClientSet().CoreV1().Nodes().
        List(context.TODO(), metav1.ListOptions{})
    if err != nil {
        return nil, err
    }

    prices := make(map[string]float64)
    for _, node := range nodes.Items {
        priceStr := node.Labels["node-price-per-hour"]
        if priceStr == "" {
            priceStr = "0.1"  // 默认价格
        }
        price, _ := strconv.ParseFloat(priceStr, 64)
        prices[node.Name] = price
    }

    return prices, nil
}
```

#### 步骤2：实现调度器主体

**pkg/scheduler/scheduler.go**：

```go
package scheduler

import (
    "context"
    "fmt"
    "time"

    v1 "k8s.io/api/core/v1"
    metav1 "k8s.io/apimachinery/pkg/apis/meta/v1"
    "k8s.io/apimachinery/pkg/util/wait"
    "k8s.io/client-go/informers"
    "k8s.io/client-go/kubernetes"
    "k8s.io/client-go/tools/cache"
    "k8s.io/klog/v2"
)

const (
    SchedulerName = "simple-scheduler"
)

// SimpleScheduler 自定义调度器
type SimpleScheduler struct {
    clientset kubernetes.Interface
    podQueue  *PodQueue
    cache     *SchedulerCache
    algorithm *SchedulingAlgorithm
}

// NewSimpleScheduler 创建调度器实例
func NewSimpleScheduler(clientset kubernetes.Interface) *SimpleScheduler {
    return &SimpleScheduler{
        clientset: clientset,
        podQueue:  NewPodQueue(),
        cache:     NewSchedulerCache(),
        algorithm: NewSchedulingAlgorithm(),
    }
}

// Run 启动调度器
func (s *SimpleScheduler) Run(ctx context.Context) {
    klog.Info("Starting Simple Scheduler")

    // 启动Informer
    informerFactory := informers.NewSharedInformerFactory(s.clientset, 0)

    // Pod Informer
    podInformer := informerFactory.Core().V1().Pods()
    podInformer.Informer().AddEventHandler(cache.ResourceEventHandlerFuncs{
        AddFunc:    s.addPodToQueue,
        UpdateFunc: s.updatePodInQueue,
        DeleteFunc: s.deletePodFromQueue,
    })

    // Node Informer
    nodeInformer := informerFactory.Core().V1().Nodes()
    nodeInformer.Informer().AddEventHandler(cache.ResourceEventHandlerFuncs{
        AddFunc:    s.addNodeToCache,
        UpdateFunc: s.updateNodeInCache,
        DeleteFunc: s.deleteNodeFromCache,
    })

    // 启动Informer
    informerFactory.Start(ctx.Done())

    // 等待缓存同步
    klog.Info("Waiting for informer caches to sync")
    if !cache.WaitForCacheSync(ctx.Done(),
        podInformer.Informer().HasSynced,
        nodeInformer.Informer().HasSynced) {
        klog.Fatal("Failed to sync informer caches")
    }

    klog.Info("Informer caches synced, starting scheduling loop")

    // 启动调度循环
    go wait.UntilWithContext(ctx, s.scheduleOne, 0)

    <-ctx.Done()
    klog.Info("Shutting down Simple Scheduler")
}

// addPodToQueue 添加Pod到队列
func (s *SimpleScheduler) addPodToQueue(obj interface{}) {
    pod := obj.(*v1.Pod)

    // 只处理使用本调度器且未调度的Pod
    if pod.Spec.SchedulerName != SchedulerName {
        return
    }

    if pod.Spec.NodeName == "" {
        klog.V(3).Infof("Adding pod %s/%s to scheduling queue",
            pod.Namespace, pod.Name)
        s.podQueue.Add(pod)
    }
}

// updatePodInQueue 更新队列中的Pod
func (s *SimpleScheduler) updatePodInQueue(oldObj, newObj interface{}) {
    oldPod := oldObj.(*v1.Pod)
    newPod := newObj.(*v1.Pod)

    if oldPod.ResourceVersion == newPod.ResourceVersion {
        return
    }

    // 如果Pod被调度，从队列移除
    if newPod.Spec.NodeName != "" {
        s.podQueue.Delete(newPod)
        return
    }

    // 如果Pod变为不可调度，添加到Backoff队列
    if newPod.Status.Phase == v1.PodFailed {
        s.podQueue.AddUnschedulable(newPod)
        return
    }

    s.podQueue.Update(newPod)
}

// deletePodFromQueue 从队列删除Pod
func (s *SimpleScheduler) deletePodFromQueue(obj interface{}) {
    pod := obj.(*v1.Pod)
    s.podQueue.Delete(pod)
}

// addNodeToCache 添加Node到缓存
func (s *SimpleScheduler) addNodeToCache(obj interface{}) {
    node := obj.(*v1.Node)
    klog.V(3).Infof("Adding node %s to cache", node.Name)
    s.cache.AddNode(node)
}

// updateNodeInCache 更新缓存中的Node
func (s *SimpleScheduler) updateNodeInCache(oldObj, newObj interface{}) {
    node := newObj.(*v1.Node)
    s.cache.UpdateNode(node)
}

// deleteNodeFromCache 从缓存删除Node
func (s *SimpleScheduler) deleteNodeFromCache(obj interface{}) {
    node := obj.(*v1.Node)
    klog.V(3).Infof("Deleting node %s from cache", node.Name)
    s.cache.DeleteNode(node)
}

// scheduleOne 调度一个Pod
func (s *SimpleScheduler) scheduleOne(ctx context.Context) {
    // 从队列获取待调度的Pod
    pod := s.podQueue.Pop()
    if pod == nil {
        return
    }

    klog.V(2).Infof("Attempting to schedule pod %s/%s",
        pod.Namespace, pod.Name)

    // 执行调度算法
    node, err := s.schedulePod(ctx, pod)
    if err != nil {
        klog.Errorf("Failed to schedule pod %s/%s: %v",
            pod.Namespace, pod.Name, err)

        // 调度失败，重新加入队列
        s.podQueue.AddUnschedulable(pod)

        // 发送事件
        s.recordSchedulingFailure(pod, err)
        return
    }

    // 绑定Pod到Node
    if err := s.bindPod(ctx, pod, node); err != nil {
        klog.Errorf("Failed to bind pod %s/%s to node %s: %v",
            pod.Namespace, pod.Name, node, err)

        // 绑定失败，重新加入队列
        s.podQueue.AddUnschedulable(pod)
        return
    }

    klog.V(2).Infof("Successfully scheduled pod %s/%s to node %s",
        pod.Namespace, pod.Name, node)

    // 发送成功事件
    s.recordSchedulingSuccess(pod, node)
}

// schedulePod 调度Pod到合适的Node
func (s *SimpleScheduler) schedulePod(ctx context.Context, pod *v1.Pod) (string, error) {
    // 获取所有可用节点
    nodes := s.cache.GetNodes()
    if len(nodes) == 0 {
        return "", fmt.Errorf("no nodes available")
    }

    klog.V(4).Infof("Scheduling pod %s/%s, evaluating %d nodes",
        pod.Namespace, pod.Name, len(nodes))

    // 阶段1: 预选（Predicate/Filter）
    feasibleNodes, err := s.algorithm.Filter(pod, nodes)
    if err != nil {
        return "", fmt.Errorf("filter phase failed: %v", err)
    }

    if len(feasibleNodes) == 0 {
        return "", fmt.Errorf("no nodes passed filter phase")
    }

    klog.V(4).Infof("Pod %s/%s: %d nodes passed filter phase",
        pod.Namespace, pod.Name, len(feasibleNodes))

    // 阶段2: 优选（Priority/Score）
    nodeScores, err := s.algorithm.Score(pod, feasibleNodes)
    if err != nil {
        return "", fmt.Errorf("score phase failed: %v", err)
    }

    // 选择得分最高的节点
    bestNode := selectBestNode(nodeScores)

    klog.V(3).Infof("Pod %s/%s: selected node %s with score %d",
        pod.Namespace, pod.Name, bestNode, nodeScores[bestNode])

    return bestNode, nil
}

// bindPod 绑定Pod到Node
func (s *SimpleScheduler) bindPod(ctx context.Context, pod *v1.Pod, nodeName string) error {
    binding := &v1.Binding{
        ObjectMeta: metav1.ObjectMeta{
            Name:      pod.Name,
            Namespace: pod.Namespace,
        },
        Target: v1.ObjectReference{
            Kind: "Node",
            Name: nodeName,
        },
    }

    err := s.clientset.CoreV1().Pods(pod.Namespace).Bind(ctx, binding, metav1.CreateOptions{})
    if err != nil {
        return fmt.Errorf("failed to bind pod: %v", err)
    }

    return nil
}

// recordSchedulingFailure 记录调度失败事件
func (s *SimpleScheduler) recordSchedulingFailure(pod *v1.Pod, err error) {
    // 简化实现：仅打印日志
    // 生产环境应使用EventRecorder
    klog.Warningf("Scheduling failed for pod %s/%s: %v",
        pod.Namespace, pod.Name, err)
}

// recordSchedulingSuccess 记录调度成功事件
func (s *SimpleScheduler) recordSchedulingSuccess(pod *v1.Pod, nodeName string) {
    klog.Infof("Successfully assigned pod %s/%s to node %s",
        pod.Namespace, pod.Name, nodeName)
}

// selectBestNode 选择得分最高的节点
func selectBestNode(scores map[string]int64) string {
    var bestNode string
    var maxScore int64 = -1

    for node, score := range scores {
        if score > maxScore {
            maxScore = score
            bestNode = node
        }
    }

    return bestNode
}
```

**pkg/scheduler/algorithm.go**：

```go
package scheduler

import (
    "fmt"

    v1 "k8s.io/api/core/v1"
    "k8s.io/apimachinery/pkg/api/resource"
    "k8s.io/klog/v2"
)

// SchedulingAlgorithm 调度算法
type SchedulingAlgorithm struct {
    predicates []PredicateFunc
    priorities []PriorityFunc
}

// PredicateFunc 预选函数（Filter）
type PredicateFunc func(pod *v1.Pod, node *v1.Node) (bool, error)

// PriorityFunc 优选函数（Score）
type PriorityFunc func(pod *v1.Pod, node *v1.Node) (int64, error)

// NewSchedulingAlgorithm 创建调度算法
func NewSchedulingAlgorithm() *SchedulingAlgorithm {
    return &SchedulingAlgorithm{
        predicates: []PredicateFunc{
            PredicateNodeUnschedulable,
            PredicateNodeResourcesFit,
            PredicateNodeSelector,
            PredicateTaintToleration,
        },
        priorities: []PriorityFunc{
            PriorityLeastRequestedResources,
            PriorityBalancedResourceAllocation,
        },
    }
}

// Filter 预选阶段：过滤不合适的节点
func (algo *SchedulingAlgorithm) Filter(pod *v1.Pod, nodes []*v1.Node) ([]*v1.Node, error) {
    feasibleNodes := []*v1.Node{}

    for _, node := range nodes {
        fits := true
        var failReason string

        // 运行所有预选函数
        for _, predicate := range algo.predicates {
            passed, err := predicate(pod, node)
            if err != nil {
                klog.Errorf("Predicate error for pod %s on node %s: %v",
                    pod.Name, node.Name, err)
                fits = false
                failReason = err.Error()
                break
            }

            if !passed {
                fits = false
                failReason = "predicate failed"
                break
            }
        }

        if fits {
            feasibleNodes = append(feasibleNodes, node)
        } else {
            klog.V(5).Infof("Node %s is not feasible for pod %s: %s",
                node.Name, pod.Name, failReason)
        }
    }

    if len(feasibleNodes) == 0 {
        return nil, fmt.Errorf("no feasible nodes found")
    }

    return feasibleNodes, nil
}

// Score 优选阶段：为节点打分
func (algo *SchedulingAlgorithm) Score(pod *v1.Pod, nodes []*v1.Node) (map[string]int64, error) {
    nodeScores := make(map[string]int64)

    for _, node := range nodes {
        var totalScore int64 = 0

        // 运行所有优选函数
        for _, priority := range algo.priorities {
            score, err := priority(pod, node)
            if err != nil {
                klog.Errorf("Priority error for pod %s on node %s: %v",
                    pod.Name, node.Name, err)
                continue
            }

            totalScore += score
        }

        // 归一化到0-100
        avgScore := totalScore / int64(len(algo.priorities))
        nodeScores[node.Name] = avgScore
    }

    return nodeScores, nil
}

// ==================== 预选函数 ====================

// PredicateNodeUnschedulable 检查节点是否可调度
func PredicateNodeUnschedulable(pod *v1.Pod, node *v1.Node) (bool, error) {
    if node.Spec.Unschedulable {
        return false, fmt.Errorf("node is unschedulable")
    }
    return true, nil
}

// PredicateNodeResourcesFit 检查节点资源是否充足
func PredicateNodeResourcesFit(pod *v1.Pod, node *v1.Node) (bool, error) {
    // 计算Pod资源请求
    podRequests := calculatePodResourceRequest(pod)

    // 获取节点可分配资源
    nodeAllocatable := node.Status.Allocatable

    // 检查CPU
    if podRequests.Cpu().Cmp(*nodeAllocatable.Cpu()) > 0 {
        return false, fmt.Errorf("insufficient CPU: requested %v, available %v",
            podRequests.Cpu(), nodeAllocatable.Cpu())
    }

    // 检查内存
    if podRequests.Memory().Cmp(*nodeAllocatable.Memory()) > 0 {
        return false, fmt.Errorf("insufficient memory: requested %v, available %v",
            podRequests.Memory(), nodeAllocatable.Memory())
    }

    return true, nil
}

// PredicateNodeSelector 检查节点选择器
func PredicateNodeSelector(pod *v1.Pod, node *v1.Node) (bool, error) {
    if pod.Spec.NodeSelector == nil || len(pod.Spec.NodeSelector) == 0 {
        return true, nil
    }

    for key, value := range pod.Spec.NodeSelector {
        nodeValue, exists := node.Labels[key]
        if !exists || nodeValue != value {
            return false, fmt.Errorf("node selector not matched: %s=%s",
                key, value)
        }
    }

    return true, nil
}

// PredicateTaintToleration 检查污点容忍
func PredicateTaintToleration(pod *v1.Pod, node *v1.Node) (bool, error) {
    // 检查节点污点
    for _, taint := range node.Spec.Taints {
        if taint.Effect == v1.TaintEffectNoSchedule ||
            taint.Effect == v1.TaintEffectNoExecute {

            // 检查Pod是否容忍此污点
            if !tolerationMatches(pod.Spec.Tolerations, &taint) {
                return false, fmt.Errorf("pod doesn't tolerate taint %s=%s:%s",
                    taint.Key, taint.Value, taint.Effect)
            }
        }
    }

    return true, nil
}

// ==================== 优选函数 ====================

// PriorityLeastRequestedResources 资源最少分配优先
func PriorityLeastRequestedResources(pod *v1.Pod, node *v1.Node) (int64, error) {
    podRequests := calculatePodResourceRequest(pod)
    allocatable := node.Status.Allocatable

    // 计算CPU使用率
    cpuUsage := float64(podRequests.Cpu().MilliValue()) /
        float64(allocatable.Cpu().MilliValue())

    // 计算内存使用率
    memUsage := float64(podRequests.Memory().Value()) /
        float64(allocatable.Memory().Value())

    // 平均使用率
    avgUsage := (cpuUsage + memUsage) / 2

    // 转换为分数（使用率越低分数越高）
    score := int64((1 - avgUsage) * 100)

    return score, nil
}

// PriorityBalancedResourceAllocation 资源平衡分配
func PriorityBalancedResourceAllocation(pod *v1.Pod, node *v1.Node) (int64, error) {
    podRequests := calculatePodResourceRequest(pod)
    allocatable := node.Status.Allocatable

    // 计算CPU和内存的使用率
    cpuUsage := float64(podRequests.Cpu().MilliValue()) /
        float64(allocatable.Cpu().MilliValue())

    memUsage := float64(podRequests.Memory().Value()) /
        float64(allocatable.Memory().Value())

    // 计算差异（越小越好）
    diff := cpuUsage - memUsage
    if diff < 0 {
        diff = -diff
    }

    // 转换为分数（差异越小分数越高）
    score := int64((1 - diff) * 100)

    return score, nil
}

// ==================== 辅助函数 ====================

// calculatePodResourceRequest 计算Pod资源请求
func calculatePodResourceRequest(pod *v1.Pod) v1.ResourceList {
    requests := v1.ResourceList{
        v1.ResourceCPU:    resource.Quantity{},
        v1.ResourceMemory: resource.Quantity{},
    }

    for _, container := range pod.Spec.Containers {
        if container.Resources.Requests != nil {
            if cpu := container.Resources.Requests[v1.ResourceCPU]; !cpu.IsZero() {
                requests[v1.ResourceCPU].Add(cpu)
            }
            if mem := container.Resources.Requests[v1.ResourceMemory]; !mem.IsZero() {
                requests[v1.ResourceMemory].Add(mem)
            }
        }
    }

    return requests
}

// tolerationMatches 检查容忍是否匹配污点
func tolerationMatches(tolerations []v1.Toleration, taint *v1.Taint) bool {
    for _, toleration := range tolerations {
        if toleration.Key == taint.Key &&
            toleration.Effect == taint.Effect &&
            (toleration.Operator == v1.TolerationOpExists ||
                toleration.Value == taint.Value) {
            return true
        }
    }
    return false
}
```

```go
    defer c.mu.RUnlock()

}
```

```go
package scheduler

import (
    "sync"
    "time"

    v1 "k8s.io/api/core/v1"
)

// PodQueue Pod调度队列
type PodQueue struct {
    mu              sync.RWMutex
    activeQ         []*v1.Pod
    backoffQ        map[string]*v1.Pod
    unschedulableQ  map[string]*v1.Pod
    backoffDuration time.Duration
}

// NewPodQueue 创建队列
func NewPodQueue() *PodQueue {
    return &PodQueue{
        activeQ:         []*v1.Pod{},
        backoffQ:        make(map[string]*v1.Pod),
        unschedulableQ:  make(map[string]*v1.Pod),
        backoffDuration: 10 * time.Second,
    }
}

// Add 添加Pod到活跃队列
func (q *PodQueue) Add(pod *v1.Pod) {
    q.mu.Lock()
    defer q.mu.Unlock()

    key := podKey(pod)

    // 从其他队列移除
    delete(q.backoffQ, key)
    delete(q.unschedulableQ, key)

    // 添加到活跃队列
    q.activeQ = append(q.activeQ, pod)
}

// Pop 从队列获取Pod
func (q *PodQueue) Pop() *v1.Pod {
    q.mu.Lock()
    defer q.mu.Unlock()

    // 尝试从backoff队列移动Pod
    q.moveBackoffToActive()

    if len(q.activeQ) == 0 {
        return nil
    }

    // 从队列头部取出
    pod := q.activeQ[0]
    q.activeQ = q.activeQ[1:]

    return pod
}

// Update 更新Pod
func (q *PodQueue) Update(pod *v1.Pod) {
    q.mu.Lock()
    defer q.mu.Unlock()

    key := podKey(pod)

    // 从所有队列删除旧版本
    q.deleteFromActiveQ(pod)
    delete(q.backoffQ, key)
    delete(q.unschedulableQ, key)

    // 添加新版本到活跃队列
    q.activeQ = append(q.activeQ, pod)
}

// Delete 删除Pod
func (q *PodQueue) Delete(pod *v1.Pod) {
    q.mu.Lock()
    defer q.mu.Unlock()

    key := podKey(pod)

    q.deleteFromActiveQ(pod)
    delete(q.backoffQ, key)
    delete(q.unschedulableQ, key)
}

// AddUnschedulable 添加到不可调度队列
func (q *PodQueue) AddUnschedulable(pod *v1.Pod) {
    q.mu.Lock()
    defer q.mu.Unlock()

    key := podKey(pod)

    // 从活跃队列移除
    q.deleteFromActiveQ(pod)

    // 添加到backoff队列（延迟重试）
    q.backoffQ[key] = pod

    // 设置定时器，延迟后移回活跃队列
    go func() {
        time.Sleep(q.backoffDuration)
        q.mu.Lock()
        defer q.mu.Unlock()

        if _, exists := q.backoffQ[key]; exists {
            delete(q.backoffQ, key)
            q.unschedulableQ[key] = pod
        }
    }()
}

// moveBackoffToActive 将backoff队列的Pod移回活跃队列
func (q *PodQueue) moveBackoffToActive() {
    for key, pod := range q.backoffQ {
        delete(q.backoffQ, key)
        q.activeQ = append(q.activeQ, pod)
    }
}

// deleteFromActiveQ 从活跃队列删除Pod
func (q *PodQueue) deleteFromActiveQ(pod *v1.Pod) {
    key := podKey(pod)
    for i, p := range q.activeQ {
        if podKey(p) == key {
            q.activeQ = append(q.activeQ[:i], q.activeQ[i+1:]...)
            return
        }
    }
}

// podKey 生成Pod的唯一键
func podKey(pod *v1.Pod) string {
    return pod.Namespace + "/" + pod.Name
}
```

```go
package main

import (
    "context"
    "flag"
    "os"
    "os/signal"
    "syscall"

    "k8s.io/client-go/kubernetes"
    "k8s.io/client-go/rest"
    "k8s.io/client-go/tools/clientcmd"
    "k8s.io/klog/v2"

    "github.com/mycompany/simple-scheduler/pkg/scheduler"
)

var (
    kubeconfig string
    masterURL  string
)

func main() {
    klog.InitFlags(nil)
    flag.StringVar(&kubeconfig, "kubeconfig", "",
        "Path to kubeconfig file (optional, default in-cluster)")
    flag.StringVar(&masterURL, "master", "",
        "URL of Kubernetes API server (optional)")
    flag.Parse()

    // 创建Kubernetes客户端
    config, err := buildConfig(kubeconfig, masterURL)
    if err != nil {
        klog.Fatalf("Failed to build config: %v", err)
    }

    clientset, err := kubernetes.NewForConfig(config)
    if err != nil {
        klog.Fatalf("Failed to create clientset: %v", err)
    }

    // 创建调度器
    sched := scheduler.NewSimpleScheduler(clientset)

    // 设置信号处理
    ctx, cancel := context.WithCancel(context.Background())
    defer cancel()

    sigCh := make(chan os.Signal, 1)
    signal.Notify(sigCh, syscall.SIGINT, syscall.SIGTERM)

    go func() {
        <-sigCh
        klog.Info("Received shutdown signal")
        cancel()
    }()

    // 启动调度器
    sched.Run(ctx)
}

// buildConfig 构建Kubernetes配置
func buildConfig(kubeconfig, masterURL string) (*rest.Config, error) {
    if kubeconfig != "" {
        // 使用kubeconfig文件
        return clientcmd.BuildConfigFromFlags(masterURL, kubeconfig)
    }

    // 使用in-cluster配置
    return rest.InClusterConfig()
}
```

```go
    // 优雅关闭
    klog.Info("Shutting down scheduler, waiting for in-flight operations")

    // 等待队列清空
    timeout := time.After(30 * time.Second)
    ticker := time.NewTicker(1 * time.Second)

        select {
        case <-timeout:

            return
        case <-ticker.C:

                klog.Info("All pods processed, exiting")
```

## 15.4 调度器插件开发

### 15.4.1 插件开发 vs 自定义调度器

**选择决策树**：

```
需要定制调度逻辑？
    │
    ├─ 是 ──► 需要完全替换调度流程？
    │         │
    │         ├─ 是 ──► 自定义调度器（15.3节）
    │         │         示例：Volcano、Kube-batch
    │         │
    │         └─ 否 ──► 插件开发（本节）✅
    │                   示例：GPU调度、数据本地化
    │
    └─ 否 ──► 使用默认调度器
```

**对比分析**：

| 维度 | 插件开发 | 自定义调度器 |
|------|---------|-------------|
| **开发复杂度** | ⭐⭐ 低 | ⭐⭐⭐⭐⭐ 高 |
| **代码量** | 100-500行 | 1000-5000行 |
| **维护成本** | ⭐⭐ 低（跟随K8s升级） | ⭐⭐⭐⭐ 高（需手动适配） |
| **功能灵活性** | ⭐⭐⭐ 中（受框架限制） | ⭐⭐⭐⭐⭐ 高（完全控制） |
| **升级兼容性** | ⭐⭐⭐⭐⭐ 好 | ⭐⭐ 差 |
| **部署方式** | 集成到kube-scheduler | 独立部署 |
| **适用场景** | 扩展现有逻辑 | 全新调度算法 |

**何时使用插件**：
- ✅ 在现有调度流程中添加自定义逻辑
- ✅ 特定扩展点（Filter、Score等）的增强
- ✅ 希望与内置插件协同工作
- ✅ 需要随K8s版本自动升级

**何时使用自定义调度器**：
- ✅ 完全不同的调度哲学（如批处理、拓扑感知）
- ✅ 需要独立的队列管理
- ✅ 需要独立部署和配置
- ✅ 特定工作负载专用

## 15.5 多调度器共存

### 15.5.1 多调度器架构概述

#### 为什么需要多调度器？

在大规模Kubernetes集群中，不同类型的工作负载可能需要不同的调度策略：

**典型场景**：

```
集群工作负载分类：
┌────────────────────────────────────────────────────────────┐
│  默认调度器 (default-scheduler)                             │
│  ├─ 通用应用                                               │
│  ├─ Web服务                                                │
│  └─ 无特殊要求的Pod                                        │
├────────────────────────────────────────────────────────────┤
│  批处理调度器 (batch-scheduler)                             │
│  ├─ 大数据作业 (Spark, Flink)                             │
│  ├─ 科学计算                                               │
│  └─ Gang调度需求                                           │
├────────────────────────────────────────────────────────────┤
│  AI训练调度器 (gpu-scheduler)                               │
│  ├─ 机器学习训练                                           │
│  ├─ GPU密集型任务                                          │
│  └─ 分布式训练 (TensorFlow, PyTorch)                      │
├────────────────────────────────────────────────────────────┤
│  实时调度器 (realtime-scheduler)                            │
│  ├─ 低延迟应用                                             │
│  ├─ 边缘计算                                               │
│  └─ IoT设备管理                                            │
└────────────────────────────────────────────────────────────┘
```

**多调度器的优势**：

| 优势 | 说明 |
|------|------|
| **职责分离** | 不同调度器专注于特定工作负载 |
| **隔离性** | 调度器故障不影响其他类型的Pod |
| **灵活性** | 可为不同租户/团队配置专用调度器 |
| **性能** | 减少单个调度器的负载 |
| **独立演进** | 各调度器可独立升级和配置 |

### 15.5.5 调度器冲突处理

#### 问题场景

**并发绑定冲突**：

```
时间线：
T0: Pod A 由 scheduler-1 选择 Node X
T1: Pod B 由 scheduler-2 也选择 Node X
T2: scheduler-1 绑定 Pod A 到 Node X (成功)
T3: scheduler-2 绑定 Pod B 到 Node X (可能失败，资源不足)
```

#### 解决方案

**1. Optimistic并发控制**

Kubernetes使用乐观锁机制处理冲突：

```go
// 调度器绑定Pod的过程
func (s *Scheduler) bindPod(pod *v1.Pod, node string) error {
    // 1. Assume: 乐观地假设绑定会成功，更新本地缓存
    err := s.Cache.AssumePod(pod, node)
    if err != nil {
        return err
    }

    // 2. Bind: 尝试实际绑定
    binding := &v1.Binding{
        ObjectMeta: metav1.ObjectMeta{
            Name:      pod.Name,
            Namespace: pod.Namespace,
        },
        Target: v1.ObjectReference{
            Kind: "Node",
            Name: node,
        },
    }

    err = s.client.CoreV1().Pods(pod.Namespace).Bind(ctx, binding, metav1.CreateOptions{})
    if err != nil {
        // 3. 绑定失败，回滚缓存
        s.Cache.ForgetPod(pod)
        return err
    }

    return nil
}
```

**2. 资源预留机制**


### 15.5.8 实战案例：混合工作负载集群

#### 场景描述

一个Kubernetes集群需要同时运行：
- 在线Web服务（低延迟）
- 批处理大数据作业（高吞吐）
- AI训练任务（GPU密集）

#### 架构设计

```
集群架构：
┌──────────────────────────────────────────────────────────┐
│  Control Plane                                           │
│  ├─ default-scheduler (在线服务)                         │
│  ├─ batch-scheduler (批处理)                             │
│  └─ gpu-scheduler (AI训练)                               │
└──────────────────────────────────────────────────────────┘
                          │
        ┌─────────────────┼─────────────────┐
        ▼                 ▼                 ▼
┌──────────────┐  ┌──────────────┐  ┌──────────────┐
│ Web Node Pool│  │ Batch Pool   │  │ GPU Pool     │
│              │  │              │  │              │
│ 10 nodes     │  │ 20 nodes     │  │ 5 GPU nodes  │
│ 32GB RAM     │  │ 128GB RAM    │  │ 256GB + 8xGPU│
│ 16 vCPU      │  │ 64 vCPU      │  │ 64 vCPU      │
└──────────────┘  └──────────────┘  └──────────────┘
```

#### 实施步骤

**1. 节点标记**

```bash
# Web节点池
kubectl label nodes node-{1..10} \
  node-role.kubernetes.io/web="" \
  workload-type=online

# 批处理节点池
kubectl label nodes node-{11..30} \
  node-role.kubernetes.io/batch="" \
  workload-type=batch

# GPU节点池
kubectl label nodes gpu-node-{1..5} \
  node-role.kubernetes.io/gpu="" \
  workload-type=ai \
  nvidia.com/gpu=true
```

**2. 部署三个调度器**

```yaml
# default-scheduler配置（已有，无需修改）

---
# batch-scheduler配置
apiVersion: v1
kind: ConfigMap
metadata:
  name: batch-scheduler-config
  namespace: kube-system
data:
  config.yaml: |
    apiVersion: kubescheduler.config.k8s.io/v1
    kind: KubeSchedulerConfiguration
    profiles:
    - schedulerName: batch-scheduler
      plugins:
        permit:
          enabled:
          - name: Coscheduling
        score:
          enabled:
          - name: NodeResourcesBalancedAllocation
            weight: 1
      pluginConfig:
      - name: NodeAffinity
        args:
          addedAffinity:
            requiredDuringSchedulingIgnoredDuringExecution:
              nodeSelectorTerms:
              - matchExpressions:
                - key: workload-type
                  operator: In
                  values: ["batch"]

---
# gpu-scheduler配置
apiVersion: v1
kind: ConfigMap

  name: gpu-scheduler-config

  config.yaml: |
    apiVersion: kubescheduler.config.k8s.io/v1

    - schedulerName: gpu-scheduler

        filter:
          enabled:
          - name: GPUDevicePlugin
        score:
          enabled:
          - name: GPUTopology
            weight: 10
      pluginConfig:
      - name: NodeAffinity
        args:
          addedAffinity:
            requiredDuringSchedulingIgnoredDuringExecution:
              nodeSelectorTerms:
              - matchExpressions:
                - key: nvidia.com/gpu
                  operator: Exists
```

### 15.5.9 故障案例与解决

#### 案例1：调度器脑裂

**现象**：
```bash
# 两个batch-scheduler都认为自己是leader
kubectl get lease -n kube-system | grep batch-scheduler

# 输出：
# batch-scheduler   batch-scheduler-abc123   ...
# batch-scheduler   batch-scheduler-def456   ...  # 错误！
```

**原因**：
- 网络分区导致lease失效
- 领导选举配置不当

**解决方案**：
```yaml
# 调整领导选举参数
containers:
- name: scheduler
  command:
  - --leader-elect-lease-duration=15s      # 减小lease时长
  - --leader-elect-renew-deadline=10s      # 减小续约期限
  - --leader-elect-retry-period=2s         # 增加重试频率
  - --leader-elect-resource-lock-name=batch-scheduler  # 唯一名称
```

---

#### 案例2：资源竞争导致频繁重调度

**现象**：
```bash
# Pod被频繁驱逐和重新调度
kubectl get events | grep -E "Evicted|FailedScheduling"
```

**原因**：
- 多个调度器竞争同一节点资源
- 资源预留不足

**解决方案**：
```yaml
# 为节点预留资源给特定调度器
apiVersion: v1
kind: Node
metadata:
  name: node1
  annotations:
    scheduler.alpha.kubernetes.io/reserved-resources: |
      {
        "default-scheduler": {"cpu": "4", "memory": "8Gi"},
        "batch-scheduler": {"cpu": "8", "memory": "16Gi"}
      }
```

---

#### 案例3：调度器配置冲突

**现象**：
```bash
# 调度器启动失败
kubectl logs -n kube-system batch-scheduler-xxx

# 错误：
# Error: failed to load config: duplicate plugin "NodeResourcesFit"
```

**原因**：
- Profile配置中插件重复注册
- 插件依赖冲突

**解决方案**：
```yaml
# 正确的配置
profiles:
- schedulerName: batch-scheduler
  plugins:
    filter:
      enabled:
      - name: NodeResourcesFit  # 只启用一次
      disabled:
      - name: NodeResourcesBalancedAllocation  # 明确禁用冲突插件
```
