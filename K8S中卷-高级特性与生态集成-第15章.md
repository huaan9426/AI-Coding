## 15.1 调度器架构深入

### 15.1.1 调度器概述

**什么是Kubernetes调度器？**

Kubernetes调度器（kube-scheduler）是控制平面的核心组件，负责为新创建的Pod选择合适的节点（Node）运行。调度决策考虑了资源需求、硬件/软件/策略约束、亲和性和反亲和性规则、数据本地性、工作负载干扰等因素。

**调度器的核心职责**：

```
调度器的三大职责：
┌────────────────────────────────────────────┐
│  1. 监听（Watch）                          │
│     - 监听API Server的Pod创建事件          │
│     - 过滤出未调度的Pod（.spec.nodeName为空）│
│     - 加入调度队列                         │
└────────────────────────────────────────────┘
                ↓
┌────────────────────────────────────────────┐
│  2. 调度（Schedule）                       │
│     - 预选（Predicates/Filter）            │
│       找出所有满足条件的节点               │
│     - 优选（Priorities/Score）             │
│       给节点打分，选择最优节点             │
│     - 绑定前检查（Reserve）                │
│       预留资源，执行插件                   │
└────────────────────────────────────────────┘
                ↓
┌────────────────────────────────────────────┐
│  3. 绑定（Bind）                           │
│     - 将Pod绑定到选定的节点                │
│     - 更新API Server中Pod的.spec.nodeName  │
│     - 通知相关组件（kubelet）              │
└────────────────────────────────────────────┘
```

**核心组件详解**：

**1. Informer机制**

```go
// Informer机制
type Scheduler struct {

// Informer工作原理
```

**2. 调度队列**

```go
// 调度队列的三个子队列
type SchedulingQueue interface {
    // ActiveQ: 活跃队列
    // - 存储待调度的Pod
    // - 按优先级排序（高优先级先调度）
    // - 使用堆（Heap）数据结构
    Pop() (*framework.QueuedPodInfo, error)

    // BackoffQ: 退避队列
    // - 存储调度失败的Pod
    // - 实现指数退避（Exponential Backoff）
    // - 重试间隔：1s, 2s, 4s, 8s, ... 最大64s
    AddUnschedulableIfNotPresent(*framework.QueuedPodInfo, int64) error

    // UnschedulableQ: 不可调度队列
    // - 存储暂时无法调度的Pod
    // - 等待集群状态变化（Node增加、Pod删除等）
    // - 定期将Pod移回ActiveQ重试
    MoveAllToActiveOrBackoffQueue(event string)
}

// 队列流程
┌────────────┐
│  新建Pod   │
└────────────┘
      ↓
┌────────────┐
│  ActiveQ   │ ← ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ┐
└────────────┘                         │
      ↓                                │
  调度尝试                             │
      ↓                                │
  成功？                               │
   ↙  ↘                                │
 Yes   No                              │
  ↓     ↓                              │
 绑定  ┌────────────┐                  │
      │  BackoffQ  │ ─ ─ ─ ─(指数退避)┘
      └────────────┘
           ↓
      (多次失败)
           ↓
      ┌────────────────┐
      │ UnschedulableQ │ ─ ─(集群变化)─ ┘
      └────────────────┘
```

**3. 调度框架（Scheduling Framework）**

调度框架是Kubernetes 1.15引入的新架构，将调度过程分为多个扩展点（Extension Points）。

```
调度框架的扩展点：
┌─────────────────────────────────────────────────────┐
│                 调度周期                             │
│                                                      │
│  ┌──────────────────────────────────────────────┐  │
│  │  1. PreFilter（预过滤）                      │  │
│  │     - 预处理Pod信息                          │  │
│  │     - 计算前置条件                           │  │
│  │     示例：计算Pod的亲和性要求                │  │
│  └──────────────────────────────────────────────┘  │
│                    ↓                                 │
│  ┌──────────────────────────────────────────────┐  │
│  │  2. Filter（过滤/预选）                      │  │
│  │     - 找出所有可以运行Pod的节点              │  │
│  │     - 并发执行，检查每个节点                 │  │
│  │     示例：检查节点资源是否足够               │  │
│  └──────────────────────────────────────────────┘  │
│                    ↓                                 │
│  ┌──────────────────────────────────────────────┐  │
│  │  3. PostFilter（后过滤）                     │  │
│  │     - 当Filter阶段没有合适节点时触发         │  │
│  │     - 尝试抢占低优先级Pod                    │  │
│  │     示例：抢占逻辑                           │  │
│  └──────────────────────────────────────────────┘  │
│                    ↓                                 │
│  ┌──────────────────────────────────────────────┐  │
│  │  4. PreScore（预打分）                       │  │
│  │     - 预处理打分信息                         │  │
│  │     - 生成可共享的状态                       │  │
│  └──────────────────────────────────────────────┘  │
│                    ↓                                 │
│  ┌──────────────────────────────────────────────┐  │
│  │  5. Score（打分/优选）                       │  │
│  │     - 给每个节点打分                         │  │
│  │     - 并发执行                               │  │
│  │     示例：资源利用率打分                     │  │
│  └──────────────────────────────────────────────┘  │
│                    ↓                                 │
│  ┌──────────────────────────────────────────────┐  │
│  │  6. NormalizeScore（归一化打分）             │  │
│  │     - 将不同插件的分数归一化到0-100          │  │
│  │     - 应用插件权重                           │  │
│  └──────────────────────────────────────────────┘  │
│                    ↓                                 │
│  ┌──────────────────────────────────────────────┐  │
│  │  7. Reserve（预留）                          │  │
│  │     - 维护运行时状态                         │  │
│  │     - 预留资源                               │  │
│  │     示例：预留GPU、RDMA等资源                │  │
│  └──────────────────────────────────────────────┘  │
│                    ↓                                 │
│  ┌──────────────────────────────────────────────┐  │
│  │  8. Permit（许可）                           │  │
│  │     - 批准或拒绝调度结果                     │  │
│  │     - 可以等待外部批准                       │  │
│  │     示例：等待资源配额检查                   │  │
│  └──────────────────────────────────────────────┘  │
│                                                      │
└─────────────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────────────┐
│                 绑定周期（异步）                     │
│                                                      │
│  ┌──────────────────────────────────────────────┐  │
│  │  9. PreBind（预绑定）                        │  │
│  │     - 绑定前的准备工作                       │  │
│  │     - 如挂载网络卷                           │  │
│  │     示例：预挂载PV                           │  │
│  └──────────────────────────────────────────────┘  │
│                    ↓                                 │
│  ┌──────────────────────────────────────────────┐  │
│  │  10. Bind（绑定）                            │  │
│  │      - 将Pod绑定到节点                       │  │
│  │      - 更新API Server                        │  │
│  └──────────────────────────────────────────────┘  │
│                    ↓                                 │
│  ┌──────────────────────────────────────────────┐  │
│  │  11. PostBind（后绑定）                      │  │
│  │      - 绑定后的清理工作                      │  │
│  │      - 发送事件通知                          │  │
│  └──────────────────────────────────────────────┘  │
│                                                      │
└─────────────────────────────────────────────────────┘
```

### 15.1.3 调度流程详解

**完整的调度流程**：

```
详细调度流程：
┌───────────────────────────────────────────────────┐
│  Step 1: 监听Pod创建事件                          │
│  ────────────────────────────────────────────     │
│  Informer监听API Server                           │
│  ↓                                                 │
│  检测到新创建的Pod（spec.nodeName为空）           │
│  ↓                                                 │
│  将Pod加入ActiveQ（按优先级排序）                 │
└───────────────────────────────────────────────────┘
                    ↓
┌───────────────────────────────────────────────────┐
│  Step 2: 从队列中取出Pod                          │
│  ────────────────────────────────────────────     │
│  从ActiveQ Pop一个最高优先级的Pod                 │
│  ↓                                                 │
│  检查Pod是否可以调度                              │
│  - 是否被删除                                     │
│  - 是否已经调度                                   │
│  - 是否在调度中                                   │
└───────────────────────────────────────────────────┘
                    ↓
┌───────────────────────────────────────────────────┐
│  Step 3: PreFilter阶段                            │
│  ────────────────────────────────────────────     │
│  执行PreFilter插件：                              │
│  ├─ NodePorts: 检查Pod需要的端口                  │
│  ├─ NodeAffinity: 解析节点亲和性要求              │
│  ├─ PodTopologySpread: 计算拓扑分布约束           │
│  └─ InterPodAffinity: 计算Pod间亲和性             │
│  ↓                                                 │
│  生成PreFilter状态（供后续插件使用）              │
└───────────────────────────────────────────────────┘
                    ↓
┌───────────────────────────────────────────────────┐
│  Step 4: Filter阶段（预选）                       │
│  ────────────────────────────────────────────     │
│  并发检查每个节点：                               │
│  ┌────────────────────────────────────────────┐  │
│  │ Node 1                                     │  │
│  │ ├─ NodeUnschedulable: 节点是否可调度      │  │
│  │ ├─ NodeResourcesFit: 资源是否足够         │  │
│  │ ├─ NodeName: 是否指定了节点名             │  │
│  │ ├─ NodePorts: 端口是否冲突                │  │
│  │ ├─ NodeAffinity: 节点亲和性是否满足       │  │
│  │ ├─ VolumeBinding: 存储卷是否可用          │  │
│  │ ├─ TaintToleration: 污点容忍是否匹配      │  │
│  │ ├─ PodTopologySpread: 拓扑分布是否满足    │  │
│  │ └─ InterPodAffinity: Pod亲和性是否满足    │  │
│  │ 结果: ✅ 通过                              │  │
│  └────────────────────────────────────────────┘  │
│  ┌────────────────────────────────────────────┐  │
│  │ Node 2                                     │  │
│  │ ├─ ... 检查所有Filter插件                 │  │
│  │ 结果: ❌ 资源不足                          │  │
│  └────────────────────────────────────────────┘  │
│  ┌────────────────────────────────────────────┐  │
│  │ Node 3                                     │  │
│  │ ├─ ... 检查所有Filter插件                 │  │
│  │ 结果: ✅ 通过                              │  │
│  └────────────────────────────────────────────┘  │
│  ↓                                                 │
│  过滤结果: [Node 1, Node 3]                       │
```

### 15.1.4 调度算法原理

**1. Filter算法（预选）**

Filter阶段使用多个插件并发检查每个节点，任何一个插件返回失败，该节点就被过滤掉。

**核心Filter插件**：

```yaml
# 1. NodeResourcesFit（资源适配）
算法: 检查节点的可分配资源是否满足Pod的资源请求

计算公式:
  NodeAllocatable = NodeCapacity - SystemReserved - KubeReserved
  PodRequest = sum(containers[].resources.requests)

  if PodRequest > (NodeAllocatable - NodeAllocated):
      return Fail
  else:
      return Success

示例:
  Node资源:
    - Capacity: CPU=8, Memory=16Gi
    - SystemReserved: CPU=0.5, Memory=1Gi
    - KubeReserved: CPU=0.5, Memory=1Gi
    - Allocatable: CPU=7, Memory=14Gi
    - Allocated: CPU=5, Memory=10Gi
    - Available: CPU=2, Memory=4Gi

  Pod请求:
    - CPU=1, Memory=2Gi

  结果: ✅ Success (1 < 2 and 2Gi < 4Gi)

---

# 2. NodeAffinity（节点亲和性）
算法: 检查Pod的节点亲和性是否匹配节点标签

匹配规则:
  requiredDuringSchedulingIgnoredDuringExecution:
    - 必须满足（硬约束）
    - 不满足则过滤该节点

  preferredDuringSchedulingIgnoredDuringExecution:
    - 尽量满足（软约束）
    - 用于打分阶段

示例:
  Pod亲和性:
    requiredDuringSchedulingIgnoredDuringExecution:
      nodeSelectorTerms:
      - matchExpressions:
        - key: disktype
          operator: In
          values: ["ssd"]

  Node标签:
    disktype: ssd

  结果: ✅ Success

---

# 3. PodTopologySpread（拓扑分布）
算法: 检查Pod在拓扑域的分布是否满足约束

计算公式:
  Skew = PodCountInDomain - MinPodCount

  if Skew > MaxSkew:
      return Fail
  else:
      return Success

示例:
  拓扑约束:
    maxSkew: 1
    topologyKey: zone
    whenUnsatisfiable: DoNotSchedule

  当前分布:
    zone-a: 2 pods
    zone-b: 1 pod
    zone-c: 1 pod

  调度到zone-a:
    Skew = 3 - 1 = 2 > 1
    结果: ❌ Fail

  调度到zone-b:
    Skew = 2 - 1 = 1 <= 1
    结果: ✅ Success

---

# 4. TaintToleration（污点容忍）
算法: 检查Pod的容忍度是否匹配节点的污点

匹配规则:
  对于节点的每个污点:
    - 如果Pod有匹配的容忍度，则忽略该污点
    - 如果没有匹配的容忍度:
      - Effect=NoSchedule: 不可调度
      - Effect=PreferNoSchedule: 尽量不调度（软约束）
      - Effect=NoExecute: 不可调度且驱逐已有Pod

示例:
  Node污点:
    - key: gpu
      value: "true"
      effect: NoSchedule

  Pod容忍度:
    - key: gpu
      operator: Equal
      value: "true"
      effect: NoSchedule

  结果: ✅ Success (容忍度匹配)

---

# 5. VolumeBinding（存储卷绑定）
算法: 检查Pod需要的PV是否可以绑定到该节点

检查项:
  1. PVC是否已绑定到PV
  2. PV的节点亲和性是否匹配当前节点
  3. 动态PV: StorageClass是否支持该节点
  4. 本地PV: 是否在该节点上

示例:
  PVC需求:
    storageClassName: local-storage
    accessMode: ReadWriteOnce

  PV (Node 1):
    nodeAffinity:
      required:
        nodeSelectorTerms:
        - matchExpressions:
          - key: kubernetes.io/hostname
            operator: In
            values: ["node-1"]

  调度到Node 1: ✅ Success
  调度到Node 2: ❌ Fail (PV不在Node 2上)

---

# 6. InterPodAffinity（Pod间亲和性）
算法: 检查Pod间的亲和性和反亲和性

匹配规则:
  requiredDuringSchedulingIgnoredDuringExecution:
    - 亲和性: 必须与匹配的Pod在同一拓扑域
    - 反亲和性: 不能与匹配的Pod在同一拓扑域

  preferredDuringSchedulingIgnoredDuringExecution:
    - 软约束，用于打分

示例:
  Pod反亲和性:
    requiredDuringSchedulingIgnoredDuringExecution:
    - labelSelector:
        matchLabels:
          app: web
      topologyKey: kubernetes.io/hostname

  Node 1上有Pod (app=web)
  Node 2上没有Pod (app=web)

  调度到Node 1: ❌ Fail (违反反亲和性)
  调度到Node 2: ✅ Success
```

**2. Score算法（优选）**

Score阶段给每个候选节点打分，选择得分最高的节点。

**核心Score插件**：

```yaml
# 1. NodeResourcesBalancedAllocation（资源均衡分配）
算法: 节点的CPU和内存使用率越接近，分数越高

计算公式:
  CPUFraction = (PodRequestCPU + NodeAllocatedCPU) / NodeAllocatableCPU
  MemoryFraction = (PodRequestMemory + NodeAllocatedMemory) / NodeAllocatableMemory

  Score = 100 - abs(CPUFraction - MemoryFraction) * 100

示例:
  Node 1:
    CPU: 50% 使用
    Memory: 40% 使用
    差值: |50% - 40%| = 10%
    分数: 100 - 10 = 90

  Node 2:
    CPU: 80% 使用
    Memory: 30% 使用
    差值: |80% - 30%| = 50%
    分数: 100 - 50 = 50

  Node 1得分更高 ✅

---

# 2. NodeResourcesLeastAllocated（最少分配）
算法: 节点空闲资源越多，分数越高

计算公式:
  CPUScore = (NodeAllocatableCPU - NodeAllocatedCPU) / NodeAllocatableCPU * 100
  MemoryScore = (NodeAllocatableMemory - NodeAllocatedMemory) / NodeAllocatableMemory * 100

  Score = (CPUScore + MemoryScore) / 2

示例:
  Node 1:
    CPU空闲: 50%
    Memory空闲: 60%
    分数: (50 + 60) / 2 = 55

  Node 2:
    CPU空闲: 70%
    Memory空闲: 80%
    分数: (70 + 80) / 2 = 75

  Node 2得分更高 ✅

---

# 3. ImageLocality（镜像本地性）
算法: 镜像已在节点上，分数越高（减少拉取时间）

计算公式:
  ImageSize = sum(已在节点的镜像大小)
  MaxImageSize = 1000Mi (配置值)

  Score = min(ImageSize / MaxImageSize * 100, 100)

示例:
  Pod镜像: nginx:1.21 (50Mi), app:1.0 (200Mi)

  Node 1:
    已有镜像: nginx:1.21 (50Mi)
    分数: 50Mi / 1000Mi * 100 = 5

  Node 2:
    已有镜像: nginx:1.21 (50Mi), app:1.0 (200Mi)
    分数: 250Mi / 1000Mi * 100 = 25

  Node 2得分更高 ✅

---

# 4. NodeAffinity（节点亲和性打分）
算法: 节点满足preferredDuringScheduling越多，分数越高

计算公式:
  Score = sum(满足的preferred条件的权重) / sum(所有preferred条件的权重) * 100

示例:
  Pod节点亲和性:
    preferredDuringSchedulingIgnoredDuringExecution:
    - weight: 50
      preference:
        matchExpressions:
        - key: disktype
          operator: In
          values: ["ssd"]
    - weight: 30
      preference:
        matchExpressions:
        - key: zone
          operator: In
          values: ["zone-a"]

  Node 1标签:
    disktype: ssd
    zone: zone-b
    满足: disktype (权重50)
    分数: 50 / 80 * 100 = 62.5

  Node 2标签:
    disktype: ssd
    zone: zone-a
    满足: disktype + zone (权重80)
    分数: 80 / 80 * 100 = 100

  Node 2得分更高 ✅

---

# 5. InterPodAffinity（Pod间亲和性打分）
算法: 根据Pod的preferred亲和性/反亲和性打分

计算公式:
  对于每个preferred规则:
    - 统计满足条件的Pod数量
    - 分数 += 权重 * 满足数量

  最终归一化到0-100

示例:
  Pod亲和性:
    preferredDuringSchedulingIgnoredDuringExecution:
    - weight: 100
      podAffinityTerm:
        labelSelector:
          matchLabels:
            app: cache
        topologyKey: kubernetes.io/hostname

  Node 1: 有2个 app=cache 的Pod
    分数: 100 * 2 = 200 (归一化后100)

  Node 2: 没有 app=cache 的Pod
    分数: 100 * 0 = 0

  Node 1得分更高 ✅

---

# 6. PodTopologySpread（拓扑分布打分）
算法: 使拓扑域的Pod分布更均衡

计算公式:
  Skew = PodCountInDomain - MinPodCount
  Score = (MaxSkew - Skew) / MaxSkew * 100

示例:
  拓扑约束:
    maxSkew: 2
    topologyKey: zone
    whenUnsatisfiable: ScheduleAnyway

  当前分布:
    zone-a: 3 pods (MinPodCount)
    zone-b: 4 pods
    zone-c: 5 pods

  调度到zone-a:
    Skew = 4 - 3 = 1
    分数: (2 - 1) / 2 * 100 = 50

  调度到zone-c:
    Skew = 6 - 3 = 3
    分数: (2 - 3) / 2 * 100 = -50 (取0)

  zone-a得分更高 ✅
```

**插件权重配置**：

```yaml
# 配置插件权重
apiVersion: kubescheduler.config.k8s.io/v1
kind: KubeSchedulerConfiguration
profiles:
- schedulerName: default-scheduler
  plugins:
    score:
      enabled:
      - name: NodeResourcesBalancedAllocation
        weight: 1
      - name: NodeResourcesLeastAllocated
        weight: 1
      - name: ImageLocality
        weight: 5  # 镜像本地性权重更高
      - name: InterPodAffinity
        weight: 2
      - name: NodeAffinity
        weight: 2
      - name: PodTopologySpread
        weight: 2
```

**最终得分计算**：

```
最终得分 =
  NodeResourcesBalancedAllocation * 1 +
  NodeResourcesLeastAllocated * 1 +
  ImageLocality * 5 +
  InterPodAffinity * 2 +
  NodeAffinity * 2 +
  PodTopologySpread * 2

归一化到0-100范围
```
